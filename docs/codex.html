Codex on Domino

A project template guide for configuring OpenAI Codex with Domino and your domain knowledge

Overview

OpenAI Codex is an agentic coding CLI built in Rust. It runs in the terminal and can take actions on your behalf: running workflows, querying systems, analyzing data, and generating outputs. It is included with ChatGPT Plus, Pro, Business, Edu, and Enterprise plans. Out of the box, it has no knowledge of Domino, your internal tooling, or your domain. This guide shows how to close that gap.

Already included: Codex is pre-installed in the Domino Standard Environment as both a VS Code extension and a CLI tool. You do not need a custom environment unless you need to pin a specific version or add additional dependencies.

Codex is also available as a macOS desktop app, a VS Code/Cursor/Windsurf IDE extension, and a cloud interface at chatgpt.com/codex. This guide focuses on the CLI, which is what runs inside Domino workspaces.

Why the Domino plugin matters. Without it, Codex has no knowledge of how Domino works. The plugin gives Codex everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why adding your own skills makes it more powerful. The plugin covers the platform. Your skills cover your domain. When you add skills for your data pipelines, internal APIs, and compliance rules, Codex stops being a generic assistant and starts understanding how your team specifically works.

Two layers, in short:

  • The Domino platform plugin: provided by Domino, covers the full platform. This is what you get out of the box.
  • Your skills: knowledge your team adds on top. Not included in the template. Explained in Part 4.

Part 1: How It Fits Together

The template ships with three files:

/mnt/code/ ← Your project (Git repo root in Domino) ├── AGENTS.md ← Domino-specific rules, paths, compliance ├── domino_project_settings.md ← Project name/workspace owner (read by MCP server) └── .codex/ └── config.toml ← MCP servers, sandbox mode, approval policy
FilePurpose
AGENTS.md Read at the start of every session. Tells Codex what it is, what it can do, and how to behave. Codex walks the directory tree from the filesystem root to the current working directory, concatenating every AGENTS.md it finds (up to a 32 KiB limit). An AGENTS.override.md at any level takes precedence over AGENTS.md at that same level. Update it as your skills evolve.
.codex/config.toml Project-level configuration in TOML format (not JSON). Declares MCP servers, sandbox mode, and approval policy. Codex also reads a user-level config at ~/.codex/config.toml.
domino_project_settings.md Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script using $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME. Users do not need to edit this file.

At workspace start, the pre-run script symlinks the Domino platform skills into .agents/skills/ from the imported plugin repo. Those are the only skills present until your team adds domain skills of their own:

/mnt/imported/code/domino-data-lab-plugin/ ← Shared plugin (not in your repo) ├── skills/ ← 20 Domino platform skills (symlinked into .agents/skills/ at start) └── mcp-servers/ ← Domino MCP server

Part 2: The Shared Domino Plugin

The plugin at /mnt/imported/code/domino-data-lab-plugin is sourced from dominodatalab/domino-claude-plugin. It provides 20 skills covering the full Domino platform: jobs, workspaces, environments, datasets, apps, model endpoints, MLflow, GenAI tracing, distributed computing, governance, and more.

It also bundles the domino_server MCP server, which gives Codex runtime tools: run_domino_job, check_domino_job_run_status, check_domino_job_run_results, list_domino_project_files, and file sync utilities.

The plugin skills load automatically when symlinked into .agents/skills/. The MCP server is started separately via .codex/config.toml. Both are wired up as part of the project template in Part 3.

Part 3: Setting Up the Codex-Domino Template and Environment

Two things need to be created once and reused across all projects: a Domino environment and a Domino project template. The environment provides the runtime: Node.js 22+, system utilities, and MCP server dependencies. The template provides the project structure: the configuration files and imported repositories that Codex needs to connect to Domino and load its skills. When a team member creates a new project, they select both; everything comes pre-loaded.

Part 3a: Setting Up the Environment

The workspace environment installs system utilities and MCP server dependencies at build time. Codex requires Node.js 22+. It is either baked into the image (for air-gapped environments) or installed fresh at workspace start via the pre-run script (for environments with internet access).

Step 1: Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions:

Option A: Open Internet

Codex is installed fresh at every workspace start by the pre-run script, meaning it is always up to date. No image rebuild needed to update. 

# ── Install system utilities ──
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*


# ── Install Node.js 22+ (required by Codex) ──
RUN curl -fsSL https://deb.nodesource.com/setup_22.x | bash - && \
    apt-get install -y nodejs && \
    rm -rf /var/lib/apt/lists/*


# ── Create directories for Codex ──
RUN mkdir -p /home/ubuntu/.codex && \
    chown -R ubuntu:ubuntu /home/ubuntu/.codex


# ── Set environment variables ──
ENV PATH="/home/ubuntu/.local/bin:$PATH"


# ── Install MCP server dependencies (required) ──
RUN pip install --no-cache-dir fastmcp requests python-dotenv


# ── Install additional packages (optional, add what your team needs) ──
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Option B: Air-Gapped / Restricted Internet

Codex is baked into the image at build time. The pre-run script detects it is already installed and skips the npm install step. To update Codex, rebuild the image.

Base image requirements: The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement. The Dockerfile above adds everything else Codex needs on top. The Dockerfile and pre-run script can be adapted to a different base user if needed.

Authentication

Codex supports two authentication methods: ChatGPT sign-in (interactive) or an OPENAI_API_KEY environment variable. For Domino workspaces, set OPENAI_API_KEY as a Domino environment variable. This is recommended for non-interactive environments and CI/CD pipelines. Codex is included with ChatGPT Plus, Pro, Business, Edu, and Enterprise plans; alternatively, it can use the OpenAI API directly.

Step 2: Add the Pre-Run Script

The pre-run script runs automatically at every workspace start. It handles five things: installing Codex if it was not baked into the image, setting up per-user persistent storage for Codex configuration and history, symlinking the plugin's skills into the project's .agents/skills/ directory, and auto-populating domino_project_settings.md from environment variables. By installing Codex in the pre-run script, it will pull the latest version on each use of the environment. By installing Codex in the environment image, it keeps the same version from when the environment was built.

Add this as the workspace pre-run script in your Domino environment or project settings:

#!/bin/bash
# start_workspace.sh: Domino pre-run script for Codex
# Handles installation, per-user persistence, and plugin discovery.
# Run once automatically at workspace start.
#
# Security: Auth credentials are scrubbed on each cold start to prevent
# token leakage via shared /mnt/artifacts. User preferences (config.toml)
# are safe to persist as they contain no secrets.

set -e

echo "=== Codex Workspace Setup ==="

# ── 1. Install Codex if not present ──
if ! command -v codex &> /dev/null; then
    echo "[!] Codex not found. Installing..."
    npm install -g @openai/codex
fi

echo "[ok] Codex version: $(codex --version 2>/dev/null || echo 'installed')"

# ── 2. Determine per-user persistent storage ──
if [ -n "$DOMINO_USER_NAME" ]; then
    PERSIST_USER="$DOMINO_USER_NAME"
else
    PERSIST_USER="$USER"
    echo "[!] DOMINO_USER_NAME not set, falling back to: $PERSIST_USER"
fi

PERSIST_DIR="/mnt/artifacts/.codex-users/${PERSIST_USER}/.codex"
mkdir -p "$PERSIST_DIR"
echo "[i] Persistent storage: $PERSIST_DIR"

# ── 3. Symlink ~/.codex to persistent storage ──
HOME_CODEX="$HOME/.codex"

if [ -d "$HOME_CODEX" ] && [ ! -L "$HOME_CODEX" ]; then
    echo "[i] Migrating existing ~/.codex to persistent storage..."
    cp -r "$HOME_CODEX/"* "$PERSIST_DIR/" 2>/dev/null || true
    rm -rf "$HOME_CODEX"
fi

[ -L "$HOME_CODEX" ] && [ ! -e "$HOME_CODEX" ] && rm -f "$HOME_CODEX"

if [ ! -L "$HOME_CODEX" ]; then
    ln -s "$PERSIST_DIR" "$HOME_CODEX"
    echo "[ok] ~/.codex symlinked to $PERSIST_DIR"
else
    echo "[ok] ~/.codex persistence already configured"
fi

# ── 4. Symlink skills from plugin ──
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

mkdir -p /mnt/code/.agents/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

echo "[ok] Registered $(ls /mnt/code/.agents/skills/ 2>/dev/null | wc -l) skills"

# ── 5. Populate domino_project_settings.md from environment ──
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Codex setup complete ==="

What the script handles

  1. Install Codex: if codex is not on the PATH, installs it via npm install -g @openai/codex. You can also install via brew install codex on macOS.
  2. Per-user persistent storage: symlinks ~/.codex to /mnt/artifacts/.codex-users/$USER/.codex so configuration and history survive workspace restarts.
  3. Symlink skills: symlinks skills from the Domino plugin into the project's .agents/skills/ directory.
  4. Populate project settings: writes $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME into domino_project_settings.md.

What gets persisted

Codex stores user configuration and history under ~/.codex/:

  • config.toml: user-level preferences, model profiles, feature flags
  • SQLite history database: conversation history (controlled by history.persistence setting: "save-all" or "none"; size cap via history.max_bytes)
  • Auth credentials: ChatGPT login token or API key reference (stored as a file in Domino since no system keyring is available)

Without persistence, this directory is lost every time a workspace stops. The pre-run script symlinks ~/.codex to /mnt/artifacts so it survives across workspace restarts, following the same pattern as the Claude Code setup.

Part 3b: Setting Up the Template

The template is a Domino project template backed by a git repository. The repository contains the three configuration files that wire Codex to the Domino platform. An additional repository is attached as Imported Code; it mounts at a fixed path that the config files reference.

Step 1: Create the Git Repository

Create a new git repository containing these three files:

/mnt/code/ ├── AGENTS.md ├── domino_project_settings.md └── .codex/ └── config.toml

Here is what each file should contain:

.codex/config.toml

Declares the Domino MCP server, sandbox mode, and approval policy. Environment variables are injected automatically by Domino. Codex uses TOML format for configuration, not JSON. 

# .codex/config.toml
# Project-level Codex configuration for Domino

# ── Domino MCP Server ──
[mcp_servers.domino_server]
command = "python"
args = ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"]

[mcp_servers.domino_server.env]
DOMINO_API_KEY = "${DOMINO_USER_API_KEY}"
DOMINO_HOST = "${DOMINO_API_HOST}"

# ── Sandbox & Approval ──
# sandbox_mode options: "read-only", "workspace-write" (default), "danger-full-access"
sandbox_mode = "workspace-write"

# approval_policy options: "untrusted", "on-request", "never"
# Granular: sandbox_approval, rules, mcp_elicitations, skill_approval
approval_policy = "on-request"

AGENTS.md

The persistent system prompt for Codex, read automatically at every session. Start with the core Domino platform rules below and extend with your organization's requirements. See Part 4 for guidance on what belongs here vs. in skills. 

# Project Instructions for Codex

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.


## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md`
to get the `project_name` and `user_name`. If they contain placeholder
values, immediately update the file with the real values from
environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`


## MCP Server

The `domino_server` MCP is configured in `.codex/config.toml` and
requires `DOMINO_API_KEY` and `DOMINO_HOST` environment variables.
Available tools:
- `run_domino_job`: run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results`:
  poll and retrieve job output
- `list_domino_project_files`: browse project files
- `sync_local_file_to_domino` / `smart_sync_file` /
  `upload_file_to_domino_project` /
  `download_file_from_domino_project`: file sync
- `get_domino_environment_info`: inspect compute environments
- `open_web_browser`: open Domino URLs


## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs
  rather than in the local terminal. This ensures governance,
  reproducibility, and audit trails.
- The Domino project name and user name are in
  domino_project_settings.md. Read this file before making any
  MCP tool calls.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.


## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.


## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.


## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

domino_project_settings.md

Tells the MCP server which Domino project to operate against. This file is auto-populated at workspace start by the pre-run script using $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME. Users do not need to edit it. 

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Part 4: Building Your Domain Skills

Out of the box, the template gives Codex knowledge of the Domino platform only. The domain layer (your workflows, your data, your internal APIs) is empty. This is where your team's work begins. The Domino plugin is the model: a git repository of Markdown skill files, mounted as imported code and symlinked at workspace start. You follow the same pattern for your domain.

Where to Put Skills

There are two scopes:

  • Project-specific skills: add to .agents/skills/ in your project git repo. They persist as long as you commit and push; a skill added in a workspace without pushing is lost when the workspace stops.
  • Org-wide skills: create a dedicated git repository (e.g. example-codex-skills), attach it to the project template as Imported Code mounted at /mnt/imported/code/example-codex-skills, and add one block to the pre-run script to symlink it alongside the Domino plugin:
# In start_workspace.sh, add after the existing plugin symlink loop
SKILLS_DIR="/mnt/imported/code/example-codex-skills"

mkdir -p /mnt/code/.agents/skills
for skill_dir in "$SKILLS_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

Every project that mounts this repo picks up your org skills automatically on workspace start, the same pattern as the Domino plugin. Update the repo once; every project sees it.

Skill Discovery Locations

Codex discovers skills from four locations, in order:

ScopePathDescription
Repository .agents/skills/ Project-specific skills committed to the repo
User ~/.agents/skills/ User-specific skills (persists via symlink)
Admin /etc/codex/skills/ System-wide skills set by administrators
System (bundled with Codex) Built-in skills shipped with the Codex binary

Anatomy of a Skill

Codex uses the Agent Skills standard. Each skill is a directory containing a SKILL.md file with YAML frontmatter: 

---
name: skill-1
description: What this skill covers and when Codex should use it.
---

# Skill-1

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages Codex should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes
Known failure modes and how to resolve them.

The description field is what Codex uses to decide when to load the skill. Write it as a specific, action-oriented sentence. Skills load only when relevant.

AGENTS.md vs. Skills

FileScopeWhen to use
AGENTS.md Universal Rules for every conversation: platform behavior, path conventions, compliance.
SKILL.md Context-specific Knowledge for a particular workflow. If a rule only applies when running a specific pipeline or tool, it belongs in a skill.

Portability: Skills using the Agent Skills standard are portable between Claude Code, Codex, and Kiro. The same SKILL.md files work across all three agents. The Domino plugin skills are shared by all of them.

Part 5: The Complete Template

Once the template is set up and the pre-run script has run, a fully built-out project looks like this:

/mnt/code/ ← Your project (Git repo root in Domino) ├── AGENTS.md ← Platform rules, data paths, compliance ├── domino_project_settings.md ← project_name and user_name ├── .codex/ │ └── config.toml ← MCP server config, sandbox, approval policy └── .agents/ └── skills/ ├── [domino platform skills, symlinked from plugin] ├── etl-pipeline/SKILL.md ← your workflow ├── data-sources/SKILL.md ← your data access patterns ├── internal-apis/SKILL.md ← your proprietary tools and APIs └── compliance/SKILL.md ← your governance and regulatory rules

The Domino plugin skills are symlinked from /mnt/imported/code/domino-data-lab-plugin/skills/ into .agents/skills/ by the pre-run script. Domain skills are added directly to the project repo or mounted from a shared org repository.

Part 6: Extending Further

Custom MCP Servers

Most teams will not need this initially; skills cover a wide range of use cases. Write a custom MCP server when you need Codex to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

Add additional [mcp_servers.<name>] sections to .codex/config.toml. Codex supports both stdio and HTTP MCP servers. All declared servers are auto-launched at session start. 

# Add to .codex/config.toml

[mcp_servers.internal_api]
command = "python"
args = [".codex/mcp-servers/internal_api.py"]

[mcp_servers.internal_api.env]
API_TOKEN = "${INTERNAL_API_TOKEN}"

A minimal MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("server-1")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def tool_1(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Key principles: one tool per action, precise docstrings (Codex reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Profiles

Define model profiles for different use cases. Profiles let you switch between models without changing config. Use profiles to balance speed vs. reasoning depth. 

# Add to .codex/config.toml or ~/.codex/config.toml

[profiles.fast]
model = "gpt-4.1-mini"

[profiles.powerful]
model = "o3"

Multi-Agent Mode

Enable multi-agent mode for complex workflows that benefit from spawning sub-agents. Each sub-agent runs in its own sandbox with focused context. 

# Add to .codex/config.toml

[features]
multi_agent = true

Custom Slash Commands

Codex supports custom slash commands. For repetitive multi-step workflows, define commands that Codex expands into full instruction sets. Good candidates: initializing new jobs from templates, scaffolding experiments with standard MLflow setup, generating data access request templates.

Non-Interactive Mode

For CI/CD pipelines and scripting, use codex exec to run Codex non-interactively. It accepts a prompt, executes it, and exits. Useful for automated code review, batch processing, and scheduled tasks. 

codex exec "Run the test suite and report failures"

Session Resume

Codex supports resuming previous sessions. This is useful when you need to pick up where you left off after a workspace restart or terminal disconnect. 

codex resume --last

Code Review

Codex includes a built-in code review capability. Use the /review command to have Codex analyze your changes and provide feedback.

Web Search and Image Input

Codex can search the web for current information and accept image files as input for visual analysis tasks. These features work out of the box with no additional configuration.

History Management

Control how Codex stores conversation history:

# Add to ~/.codex/config.toml (user-level)

[history]
persistence = "save-all"   # or "none" to disable
max_bytes = 104857600      # 100 MB cap

Custom Themes

Codex supports custom themes to match your terminal color scheme. Configure themes in the user-level config at ~/.codex/config.toml.

AGENTS.md vs. CLAUDE.md: Key Differences

If your team runs both Codex and Claude Code, you will have two system prompt files. They serve the same purpose but use different conventions:

FeatureClaude CodeCodex
System prompt file CLAUDE.md AGENTS.md
Override file (none) AGENTS.override.md
Global instructions ~/.claude/CLAUDE.md ~/.codex/AGENTS.md
Config format JSON (.mcp.json, settings.json) TOML (.codex/config.toml)
Skills directory .claude/skills/ .agents/skills/
Skill file format SKILL.md (Agent Skills standard) SKILL.md (Agent Skills standard)
MCP config .mcp.json (JSON) .codex/config.toml (TOML sections)
Size limit (no documented limit) 32 KiB (concatenated across directories)
Directory walking Reads project root Walks root to CWD, concatenating all found

The system prompt content is nearly identical for both. The Domino platform rules, data access patterns, and Git discipline sections are shared. The only structural difference is the config file format and skill directory path.
