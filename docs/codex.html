Codex on Domino

Agent-specific configuration for OpenAI Codex

OpenAI Codex is an agentic coding CLI that runs in the terminal. It reads an AGENTS.md system prompt, connects to MCP servers defined in .codex/config.toml, and discovers skills from .agents/skills/. This page covers the Codex-specific configuration files and setup steps needed to run Codex inside a Domino workspace. For shared concepts (environments, the Domino plugin, domain skills), see the Overview page.

Part 1: Template Files

The Codex template ships with three files:

/mnt/code/ ├── AGENTS.md ← System prompt (read every session) ├── domino_project_settings.md ← Project name/user (shared) └── .codex/ └── config.toml ← MCP servers, sandbox, permissions

AGENTS.md is the equivalent of Claude Code's CLAUDE.md. Codex walks the directory tree from the filesystem root to the current working directory, concatenating every AGENTS.md it finds (up to a 32 KiB limit). An AGENTS.override.md file at any level takes precedence over AGENTS.md at that same level.

.codex/config.toml is the Codex configuration file (TOML format, not JSON). It declares MCP servers, sandbox mode, and approval policy.

domino_project_settings.md is the same shared file used by all agents. It tells the MCP server which Domino project to operate against.

Part 2: Environment Setup

Codex requires Node.js 22+. The Domino environment must provide it.

Option A: Open Internet

Install Codex fresh at every workspace start via the pre-run script. Always up to date; no image rebuild needed.

# Add to Dockerfile instructions (after Node.js 22+ is available)

# ── Install MCP server dependencies (required) ──
USER root
RUN pip install --no-cache-dir fastmcp requests python-dotenv
USER ubuntu

The pre-run script (below) runs npm install -g @openai/codex on each workspace start.

Option B: Air-Gapped / Restricted Internet

Bake Codex into the image at build time. To update, rebuild the image.

# ── Install Codex into the image ──
USER root
RUN npm install -g @openai/codex
USER ubuntu

# ── Install MCP server dependencies (required) ──
USER root
RUN pip install --no-cache-dir fastmcp requests python-dotenv
USER ubuntu

Authentication

Set OPENAI_API_KEY as a Domino environment variable, or sign in with your ChatGPT account at first run. The API key approach is recommended for non-interactive environments.

Part 3: Pre-Run Script

The pre-run script is simpler than Claude Code's because Codex has no ~/.claude persistence layer to manage. It handles three things: installing Codex if not present, symlinking skills from the plugin, and populating domino_project_settings.md. 

#!/bin/bash
# start_workspace.sh — Domino pre-run script for Codex
# Run once automatically at workspace start.

set -e

echo "=== Codex Workspace Setup ==="

# ── 1. Install Codex if not present ──
if ! command -v codex &> /dev/null; then
    echo "[!] Codex not found. Installing..."
    npm install -g @openai/codex
fi
echo "[ok] Codex: $(codex --version 2>/dev/null || echo 'installed')"

# ── 2. Symlink skills from plugin ──
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"
mkdir -p /mnt/code/.agents/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done
echo "[ok] Registered $(ls /mnt/code/.agents/skills/ 2>/dev/null | wc -l) skills"

# ── 3. Populate domino_project_settings.md from environment ──
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Codex setup complete ==="

Part 4: Template File Contents

.codex/config.toml

Declares the Domino MCP server, sandbox mode, and approval policy. Environment variables are injected automatically by Domino. 

# Domino MCP Server
[mcp_servers.domino_server]
command = "python"
args = ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"]

[mcp_servers.domino_server.env]
DOMINO_API_KEY = "${DOMINO_USER_API_KEY}"
DOMINO_HOST = "${DOMINO_API_HOST}"

# Sandbox & Approval
sandbox_mode = "workspace-write"
approval_policy = "on-request"

AGENTS.md

The persistent system prompt for Codex. Same Domino platform rules as CLAUDE.md, adapted for Codex's configuration paths. 

# Project Instructions for Codex

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.


## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md`
to get the `project_name` and `user_name`. If they contain placeholder
values, immediately update the file with the real values from
environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`


## MCP Server

The `domino_server` MCP is configured in `.codex/config.toml` and
requires `DOMINO_API_KEY` and `DOMINO_HOST` environment variables.
Available tools:
- `run_domino_job` — run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results`
- `list_domino_project_files` — browse project files
- `sync_local_file_to_domino` / `smart_sync_file`
- `upload_file_to_domino_project` / `download_file_from_domino_project`
- `get_domino_environment_info` — inspect compute environments
- `open_web_browser` — open Domino URLs


## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs
  rather than in the local terminal. This ensures governance,
  reproducibility, and audit trails.
- The Domino project name and user name are in
  domino_project_settings.md. Read this file before making any
  MCP tool calls.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.


## Data Access

- Project data is accessible under /mnt/data/ or /mnt/imported/data/.
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.


## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.


## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. Users do not edit this file.

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Part 5: Skills Format

Codex uses the Agent Skills standard. Skills live in .agents/skills/*/SKILL.md and use the same format as Claude Code:

.agents/skills/ ├── jobs/SKILL.md ├── workspaces/SKILL.md ├── environments/SKILL.md ├── datasets/SKILL.md ├── app-deployment/SKILL.md ├── model-endpoints/SKILL.md ├── experiment-tracking/SKILL.md └── ... (20 Domino platform skills)

Each SKILL.md has YAML frontmatter with name and description fields:

---
name: jobs
description: Run and manage Domino Jobs for batch execution.
---

# Jobs

## Overview
Create, run, and manage Domino Jobs...

## Key tools
- `run_domino_job`
- `check_domino_job_run_status`

## Rules
- Always commit and push before running a job
- Check status and results after submission

Codex discovers skills from three locations:

  • Repository: .agents/skills/ in the project root
  • User: ~/.agents/skills/
  • Admin: /etc/codex/skills/

Portability: Skills using the Agent Skills standard are portable between Claude Code, Codex, and Kiro. The same SKILL.md files work across all three agents.

Part 6: The Complete Template

After the pre-run script has run, a fully built-out project looks like this:

/mnt/code/ ← Your project (Git repo root in Domino) ├── AGENTS.md ← Platform rules, data paths, compliance ├── domino_project_settings.md ← project_name and user_name ├── .codex/ │ └── config.toml ← MCP server config └── .agents/ └── skills/ ├── [domino platform skills, symlinked from plugin] ├── etl-pipeline/SKILL.md ← your workflow ├── data-sources/SKILL.md ← your data access patterns ├── internal-apis/SKILL.md ← your proprietary tools and APIs └── compliance/SKILL.md ← your governance and regulatory rules

The Domino plugin skills are symlinked from /mnt/imported/code/domino-data-lab-plugin/skills/ into .agents/skills/ by the pre-run script. Domain skills are added directly to the project repo or mounted from a shared org repository.

Extending

Custom MCP Servers

Add additional [mcp_servers.server_name] sections to .codex/config.toml:

[mcp_servers.internal_api]
command = "python"
args = [".codex/mcp-servers/internal_api.py"]

[mcp_servers.internal_api.env]
API_TOKEN = "${INTERNAL_API_TOKEN}"

Profiles

Define model profiles for different use cases:

[profiles.fast]
model = "gpt-4.1-mini"

[profiles.powerful]
model = "o3"

Multi-Agent

Enable multi-agent mode for complex workflows:

[features]
multi_agent = true

Verification

After setup, open a terminal in your Domino workspace and run:

codex

Then ask:

"What Domino skills do you have?"

Codex should list the Domino platform skills from the plugin. If it does, the template is working correctly.
