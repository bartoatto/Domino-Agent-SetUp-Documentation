Domino Cursor Setup Guide

A project template guide for configuring Cursor with Domino and your domain knowledge

Overview

Cursor is an AI code editor built as a fork of VS Code. Unlike CLI-based agents such as Claude Code or Codex, Cursor is an IDE — it provides AI assistance directly inside the editing environment through its Agent mode, autocomplete, and inline chat. It uses .cursor/rules/*.mdc files for persistent instructions and supports MCP servers via .cursor/mcp.json.

Cursor does not need to be installed in the Domino environment. It is a desktop application that connects to Domino workspaces remotely (via SSH or the workspace URL). The Domino environment only needs the MCP server dependencies so that Cursor's agent can interact with the Domino platform at runtime.

Why the Domino plugin matters

Without it, Cursor has no knowledge of how Domino works. The plugin's MCP server gives it everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why rules matter

The MCP server provides tools — the ability to take actions. Rules provide knowledge — how and when to use those tools, what conventions to follow, and what constraints to respect. Without rules, Cursor has tools but no understanding of your platform or domain. With rules, it becomes a context-aware assistant that follows your team's practices.

Two layers

Layer Source What it covers
Domino platform plugin Provided by Domino (MCP server + rules) Jobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing
Your domain rules Added by your team Your workflows, data pipelines, internal APIs, compliance rules

Note: The Domino plugin's skills are written in the Agent Skills standard (SKILL.md format), which is native to Claude Code, Codex, and Kiro. For Cursor, these skills need to be adapted as .mdc rule files since Cursor uses its own format with YAML frontmatter. The MCP server, however, is agent-agnostic and directly reusable.

Part 1: How It Fits Together

The Cursor-Domino integration uses three configuration files that live in your project repository:

/mnt/code/ ├── domino_project_settings.md ← Project name/user (read by MCP server) └── .cursor/ ├── rules/ │ └── domino.mdc ← Domino platform rules (always applied) └── mcp.json ← MCP server config

Each file has a specific role:

  • domino_project_settings.md — Contains the project name and username. Read by the MCP server to know which Domino project to target. Auto-populated at workspace start.
  • .cursor/mcp.json — Tells Cursor how to start the Domino MCP server. This gives Cursor's agent the ability to run jobs, sync files, and interact with Domino.
  • .cursor/rules/domino.mdc — The Domino platform rules, formatted as a Cursor rule. This tells the agent how to use those tools: check for uncommitted changes before running jobs, always poll for results, use /mnt/data/ for datasets, and so on.

When a user opens Agent mode in Cursor (Cmd+L or Ctrl+L), the agent reads the alwaysApply: true rules automatically, discovers the MCP server tools, and is ready to operate on Domino.

Key differences from Claude Code

Feature Claude Code Cursor
Runtime CLI in terminal IDE (VS Code fork)
System prompt CLAUDE.md (Markdown) .cursor/rules/*.mdc (Markdown + YAML frontmatter)
MCP config file .mcp.json .cursor/mcp.json
MCP root key mcpServers mcpServers (same)
Skills format SKILL.md (Agent Skills standard) .mdc files with frontmatter
Conditional loading Description-based Globs, description, alwaysApply, or manual
Plugin system settings.json plugins array None (rules + MCP only)
Slash commands .claude/commands/*.md Not supported in the same way

Part 2: The Shared Domino Plugin

The Domino plugin lives at /mnt/imported/code/domino-data-lab-plugin, imported into every project via Domino's Imported Code Repositories feature. It provides:

  • An MCP server — directly reusable with Cursor. The MCP server is agent-agnostic; the same Python process serves tools to any agent that can speak the MCP protocol.
  • Skills in Agent Skills format — written as SKILL.md files. These need format conversion to .mdc for Cursor (see Part 4).

MCP server tools

The MCP server provides the following tools that Cursor's agent can call:

Tool Purpose
run_domino_job Run a script as a Domino job
check_domino_job_run_status Poll a running job for its current state
check_domino_job_run_results Retrieve stdout/stderr from a completed job
list_domino_project_files Browse project files on the Domino server
sync_local_file_to_domino Push a local file to the Domino project
smart_sync_file Intelligent sync with conflict detection
upload_file_to_domino_project Upload a file to Domino project storage
download_file_from_domino_project Download a file from Domino project storage
get_domino_environment_info Inspect compute environments
open_web_browser Open Domino URLs in the browser

The plugin repository is the same one used by Claude Code, Codex, Kiro, and Copilot. The MCP server code is identical across all agents — only the configuration file that points to it differs.

Part 3: Setting Up the Cursor-Domino Template and Environment

Part 3a: Environment

Because Cursor is an IDE (not a CLI tool), the Domino environment does not need to install Cursor itself. Cursor runs as the user's local desktop application and connects to the Domino workspace remotely. The environment only needs:

  • MCP server dependencies — so the Domino MCP server can run when Cursor connects
  • System utilities — standard tools used by MCP server operations

Step 1: Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions:

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*

# -- Install MCP server dependencies (required) --
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional -- add what your team needs) --
RUN pip install dash pandas pyarrow streamlit
USER ubuntu

Note: No Cursor binary is installed in the environment. The Dockerfile is simpler than for CLI agents because there is no agent runtime to install or persist.

Step 2: Add the Pre-Run Script

The pre-run script for Cursor is simpler than for CLI agents — no agent installation, no persistent storage for authentication, no symlinked sessions. It only needs to:

  1. Populate domino_project_settings.md from environment variables
  2. Ensure .cursor/ config files are in place
  3. Optionally convert and symlink skills from the imported plugin
#!/bin/bash
# start_workspace.sh -- Domino pre-run script for Cursor
# Populates project settings and ensures config files are in place.

set -e

echo "=== Cursor Workspace Setup ==="

# ---------------------------------------------------------------
# 1. Populate domino_project_settings.md from environment
# ---------------------------------------------------------------
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

# ---------------------------------------------------------------
# 2. Ensure .cursor/ config directory exists
# ---------------------------------------------------------------
mkdir -p /mnt/code/.cursor/rules

# ---------------------------------------------------------------
# 3. Symlink org-wide Cursor rules from imported code (optional)
# ---------------------------------------------------------------
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"
if [ -d "$PLUGIN_DIR/cursor-rules" ]; then
    for rule_file in "$PLUGIN_DIR"/cursor-rules/*.mdc; do
        [ -f "$rule_file" ] || continue
        ln -sf "$rule_file" "/mnt/code/.cursor/rules/$(basename "$rule_file")" 2>/dev/null || true
    done
    echo "[ok] Symlinked Cursor rules from plugin"
fi

echo "=== Cursor workspace setup complete ==="

Part 3b: Template

The template is a Git repository containing the Cursor configuration files. When a team member creates a new Domino project from the template, they get the full Cursor-Domino integration pre-loaded.

Step 1: Create the Git Repository

Create a new Git repository with the following files:

/mnt/code/ ├── domino_project_settings.md ← Auto-populated at workspace start └── .cursor/ ├── mcp.json ← MCP server config └── rules/ └── domino.mdc ← Domino platform rules

.cursor/mcp.json

Tells Cursor how to start the Domino MCP server. Uses the same mcpServers root key as Claude Code:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

.cursor/rules/domino.mdc

The Domino platform rules, formatted as a Cursor rule with YAML frontmatter. Setting alwaysApply: true ensures it is included in every conversation:

---
description: Domino Data Lab platform rules for running jobs, accessing data, and deploying apps.
alwaysApply: true
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.cursor/mcp.json`. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve job output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` / `download_file_from_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. Users do not need to edit it:

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Cursor Rule Types

Cursor rules use YAML frontmatter to control when they are loaded. Understanding the four types is important for structuring your Domino rules and domain skills:

Type Frontmatter When loaded
Always alwaysApply: true Every conversation, automatically
Auto Attached globs: "*.py" When referenced files match the glob pattern
Agent Requested description: "..." (no globs, not alwaysApply) Agent decides based on description match
Manual Neither globs nor alwaysApply Only when user types @ruleName

For Domino, use Always for the core platform rules (domino.mdc) so they are present in every conversation. Use Agent Requested (description-based) or Auto Attached (glob-based) for domain-specific skills that should only load when relevant.

Part 4: Building Your Domain Skills

Out of the box, the template gives Cursor knowledge of the Domino platform only. The domain layer — your workflows, your data, your internal APIs — is empty. This is where your team's work begins.

Cursor Rule Format

Each domain skill is a separate .mdc file in .cursor/rules/. The format uses YAML frontmatter followed by Markdown content: 

---
description: Rules for running ETL pipelines on Domino using Spark and the data warehouse.
globs: "pipelines/**/*.py"
---

# ETL Pipeline Rules

## Overview
Brief description of what this skill covers.

## Key tools
List the MCP tools or packages the agent should use.

## How to run the pipeline
Step-by-step instructions or code examples.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints

Converting Agent Skills to Cursor Rules

The Domino plugin and many org-wide skill repositories use the Agent Skills standard (SKILL.md format), which is native to Claude Code, Codex, and Kiro. To use these skills with Cursor, convert them to .mdc format:

Agent Skills SKILL.md:

---
name: etl-pipeline
description: Rules for running ETL pipelines on Domino.
---
[body content]

Becomes Cursor .mdc:

---
description: Rules for running ETL pipelines on Domino.
alwaysApply: false
---
[same body content]

The conversion rules are straightforward:

  • The name field is dropped — Cursor uses the filename instead.
  • The description field is preserved — this is what the agent uses to decide when to load the rule.
  • Add globs if the skill applies to specific file patterns (e.g., "pipelines/**/*.py").
  • Set alwaysApply: true only for rules that should be in every conversation.

Where to Put Rules

Project-specific rules: Add directly to .cursor/rules/ in your project Git repo. Commit and push; rules persist as long as they are in the repo.

Org-wide rules: Create a dedicated Git repository (e.g., my-org-cursor-rules), attach it to the project template as Imported Code, and add a block to the pre-run script to symlink rules at workspace start:

# In start_workspace.sh, add:
ORG_RULES="/mnt/imported/code/my-org-cursor-rules"

for rule_file in "$ORG_RULES"/rules/*.mdc; do
    [ -f "$rule_file" ] || continue
    ln -sf "$rule_file" "/mnt/code/.cursor/rules/$(basename "$rule_file")" 2>/dev/null || true
done

Every project that mounts this repo picks up your org rules automatically on workspace start.

Always-On Rules vs. On-Demand Rules

Always-on rules (alwaysApply: true): Universal rules for every conversation — platform behavior, path conventions, compliance constraints. The domino.mdc file is the canonical example.

On-demand rules (globs or description-based): Context-specific knowledge for a particular workflow. If a rule only applies when working with a specific pipeline, data format, or tool, make it on-demand. This keeps conversations focused and avoids overwhelming the agent with irrelevant context.

Part 5: The Complete Template

With both the Domino platform rules and your domain skills in place, the complete project template looks like this:

/mnt/code/ ├── domino_project_settings.md ← Project name/user (auto-populated) └── .cursor/ ├── mcp.json ← MCP server config └── rules/ ├── domino.mdc ← Platform rules (always applied) ├── etl-pipeline.mdc ← Your ETL workflow (globs: "pipelines/**/*.py") ├── data-sources.mdc ← Your data access patterns (agent requested) └── compliance.mdc ← Your governance rules (always applied)

The rule loading behavior for each file:

File Type When loaded
domino.mdc Always Every conversation — core Domino platform behavior
compliance.mdc Always Every conversation — governance constraints must always apply
etl-pipeline.mdc Auto Attached When the user references files matching pipelines/**/*.py
data-sources.mdc Agent Requested When the agent determines the description matches the current task

Add more .mdc files as your team identifies new domain areas. Each file is self-contained and independently loadable.

Part 6: Extending Further

Custom MCP Servers

Most teams will not need this initially; rules cover a wide range of use cases. Write a custom MCP server when you need the agent to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

The MCP server code is the same FastMCP Python code used by any agent. Register it in .cursor/mcp.json alongside the Domino server:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "my_custom_server": {
      "command": "python",
      "args": ["/mnt/code/mcp-servers/my_custom_server.py"],
      "env": {
        "MY_API_TOKEN": "${MY_API_TOKEN}"
      }
    }
  }
}

A minimal custom MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("my-custom-server")

@mcp.tool()
def query_data_catalog(query: str) -> dict:
    """Search the internal data catalog. Returns matching datasets with descriptions."""
    resp = requests.post(
        "https://data-catalog.myorg.net/api/search",
        json={"query": query},
        headers={"Authorization": f"Bearer {os.environ['MY_API_TOKEN']}"},
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Key principles: one tool per action, precise docstrings (Cursor reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Slash Commands and Agent Requested Rules

Cursor does not have a direct equivalent of Claude Code's slash commands (.claude/commands/*.md). Instead, use Agent Requested rules to achieve a similar effect. Write a rule with a clear, action-oriented description, and the agent will load it when the user's request matches: 

---
description: Initialize a new ML training job with standard MLflow experiment tracking setup.
alwaysApply: false
---

# New Training Job Setup

When the user asks to set up a new training job:

1. Create a training script with MLflow auto-logging configured
2. Set the experiment name to match the project name
3. Commit and push the script
4. Run it as a Domino job
5. Share the MLflow experiment URL from the results

When the user says something like "set up a new training job," Cursor's agent matches the description and loads the rule automatically.
