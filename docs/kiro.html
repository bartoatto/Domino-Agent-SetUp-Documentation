Kiro on Domino

Agent-specific configuration for Kiro

Kiro is Amazon's AI IDE and CLI for agentic development. It connects to Domino through the same shared plugin and MCP server described on the Overview page. This guide covers the Kiro-specific configuration files that wire everything together.

Two context systems. Kiro has both steering files (.kiro/steering/*.md) for always-on rules and Agent Skills (.kiro/skills/) for on-demand knowledge. Steering files control platform behavior; skills provide workflow-specific context loaded progressively.

Template Files

The Kiro integration adds these files to the shared template:

/mnt/code/ ├── AGENTS.md ← System prompt (Kiro reads this standard) ├── domino_project_settings.md ← Project name/user (shared) └── .kiro/ ├── steering/ │ └── domino.md ← Domino platform rules (always loaded) ├── skills/ ← Symlinked from Domino plugin └── settings/ └── mcp.json ← MCP server config

Environment Setup

Kiro is a desktop IDE, it does not need to be installed in the Domino environment. Only the MCP server dependencies are required (covered in the Overview environment setup).

The Kiro CLI can optionally be installed in the environment for terminal-based use. See Kiro CLI documentation for installation instructions.

No Dockerfile additions needed. The common Dockerfile base from the Overview page includes everything Kiro requires. No agent-specific packages are necessary.

Pre-Run Script

The pre-run script runs at workspace start. It symlinks the Domino plugin skills into .kiro/skills/ and populates domino_project_settings.md from environment variables.

#!/bin/bash
set -e
echo "=== Kiro Workspace Setup ==="

# Symlink skills from plugin
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"
mkdir -p /mnt/code/.kiro/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done
echo "[ok] Registered $(ls /mnt/code/.kiro/skills/ 2>/dev/null | wc -l) skills"

# Populate domino_project_settings.md
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Kiro setup complete ==="

Template File Contents

.kiro/settings/mcp.json

Starts the Domino MCP server. Uses the mcpServers root key. Environment variables are injected automatically by Domino.

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

.kiro/steering/domino.md

The primary steering file for Domino platform rules. The inclusion: always frontmatter ensures it is loaded in every interaction.

---
inclusion: always
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md`
to get the `project_name` and `user_name`. If they contain placeholder
values, immediately update the file with the real values from
environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.kiro/settings/mcp.json`
and requires `DOMINO_API_KEY` and `DOMINO_HOST` environment variables.
Available tools: run_domino_job, check_domino_job_run_status,
check_domino_job_run_results, list_domino_project_files, file sync
utilities, get_domino_environment_info, open_web_browser.

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs
  rather than in the local terminal.
- When running a job, always check its status and results if completed.
- If a job result includes an MLflow or experiment run URL, share it.

## Data Access

- Project data is under /mnt/data/ or /mnt/imported/data/.
- Before using a dataset, run a job to list folder contents recursively.
- Analytical outputs should be in plain text tabular format to stdout.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- Commit and push before running jobs (Domino uses remote repo state).

## App Development

- Bind to 0.0.0.0 (not localhost) for Domino apps.
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

AGENTS.md

Kiro also reads AGENTS.md at the repository root. This can contain the same content as the steering file or serve as a simpler fallback. It follows the same standard read by Claude Code and Codex, making it portable across agents.

Steering Inclusion Modes

Kiro steering files use YAML frontmatter to control when they are loaded. The inclusion field determines the loading strategy.

Mode Frontmatter When loaded
Always inclusion: always Every interaction
File Match inclusion: fileMatch + fileMatchPattern: "*.py" When active files match the glob pattern
Manual inclusion: manual Only when explicitly referenced with #steering-name
Auto inclusion: auto + name + description Agent decides based on the description

Foundation steering files. Kiro supports optional foundation steering files for broader project context: product.md (product context), tech.md (tech stack), and structure.md (project structure). These complement the Domino-specific domino.md steering file.

Skills Format

Kiro has two context systems that serve different purposes:

  • Steering files (.kiro/steering/), always-on rules and context. Best for platform behavior, coding standards, and compliance rules that apply to every interaction.
  • Agent Skills (.kiro/skills/), on-demand workflow knowledge. Best for specific pipelines, data access patterns, and domain procedures that are only relevant sometimes.

Agent Skills follow the same SKILL.md standard used by Claude Code and Codex. This makes skills portable between these three agents with no changes required.

How skills load

  • Skills are loaded progressively: Kiro reads the metadata (name, description) at startup and loads the full content only when relevant.
  • Skills can be invoked explicitly in chat with /skill-name.

Skill file anatomy

---
name: etl-pipeline
description: Run and monitor the nightly ETL pipeline on Domino.
---

# ETL Pipeline

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use.

## How to run
Step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints

Complete Template

Once the template is set up and the pre-run script has run, a fully built-out Kiro project looks like this:

/mnt/code/ ├── AGENTS.md ├── domino_project_settings.md └── .kiro/ ├── steering/ │ ├── domino.md ← Platform rules (always) │ ├── product.md ← Product context (optional) │ └── tech.md ← Tech stack (optional) ├── skills/ │ ├── [platform skills, symlinked] │ ├── etl-pipeline/SKILL.md │ └── compliance/SKILL.md └── settings/ └── mcp.json

Extending

Custom MCP servers

Add additional MCP servers to .kiro/settings/mcp.json alongside the Domino server. The format is the same, add another entry under mcpServers.

Agent hooks

Kiro supports event-driven agent hooks that trigger on specific events: file saved, prompt submitted, response received, and more. Use hooks to automate repetitive tasks like linting on save or pre-populating context.

Explicit skill invocation

Invoke any registered skill directly in chat with /skill-name. This is useful for triggering a specific workflow without waiting for Kiro to auto-detect relevance.

Verification

Open the project in Kiro and ask:

"What Domino skills do you have?"

Kiro should respond with the Domino platform skills from the plugin. If it does, the template is working correctly. When team members create new projects from the template, domino_project_settings.md is populated automatically at workspace start, they only need to add their domain skills to .kiro/skills/ and steering files to .kiro/steering/.
