Kiro on Domino

Agent-specific configuration for Kiro

Kiro is Amazon's agentic IDE for spec-driven development. It connects to your local workspace or runs as a standalone CLI, and it can take actions on your behalf: generating code from structured specifications, running workflows, managing files, and interacting with external systems through MCP. Out of the box, it has no knowledge of Domino, your internal tooling, or your domain. This page shows how to close that gap.

Kiro is available as a desktop IDE (download from kiro.dev) and as a CLI (curl -fsSL https://cli.kiro.dev/install | bash). The IDE is supported on macOS and Linux (AppImage and Ubuntu). The CLI is supported on macOS and Linux.

Note: The shared setup, the Domino plugin, common Dockerfile dependencies, and the project template concept, is covered on the Overview page. This page focuses on the Kiro-specific configuration.

Three context systems. Kiro has three distinct ways to provide context to the agent: steering files (.kiro/steering/*.md) for persistent project guidance with inclusion modes, Agent Skills (.kiro/skills/) for on-demand workflow knowledge using the shared Agent Skills standard, and AGENTS.md at the repository root, a standard format that is always loaded with no inclusion modes. All three work together and serve different purposes.

Template Files

The project template ships with these files that wire Kiro to the Domino platform:

/mnt/code/ ├── AGENTS.md ← System prompt (always loaded, standard format) ├── domino_project_settings.md ← Project name/user (shared) └── .kiro/ ├── steering/ │ └── domino.md ← Domino platform rules (always loaded) ├── skills/ ← Symlinked from Domino plugin at workspace start └── settings/ └── mcp.json ← MCP server config
FilePurpose
AGENTS.md Read automatically at the start of every Kiro session. Follows the standard format also read by Claude Code and Codex, making it portable across agents. Always loaded, no inclusion modes.
.kiro/steering/domino.md The primary steering file for Domino platform rules. Uses inclusion: always frontmatter to ensure it is loaded in every interaction. Tells Kiro what it is, what it can do, and how to behave on Domino.
.kiro/settings/mcp.json Starts the Domino MCP server at workspace launch. Uses the mcpServers root key. Provides runtime tools like run_domino_job, check_domino_job_run_status, file sync utilities, and more.
domino_project_settings.md Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script using $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME.

Environment Setup

Kiro is a desktop IDE. It does not need to be installed in the Domino environment. Users run Kiro locally on their machine and connect to the Domino workspace remotely. Only the MCP server dependencies are required in the environment (covered in the Overview environment setup).

Kiro CLI (Optional)

The Kiro CLI can optionally be installed in the Domino environment for terminal-based use inside workspaces. This is useful for teams that prefer working in the terminal or need to run Kiro in headless mode. 

# Install Kiro CLI (optional, in pre-run script or Dockerfile)
curl -fsSL https://cli.kiro.dev/install | bash

The CLI provides interactive chat, custom agents, MCP integration, smart hooks, steering file support, and auto-complete. Authentication is handled via https://app.kiro.dev.

Pre-Run Script

The pre-run script runs automatically at every workspace start. It handles three things: symlinking the Domino plugin skills into .kiro/skills/, populating domino_project_settings.md from environment variables, and optionally installing the Kiro CLI. Add this as the workspace pre-run script in your Domino environment or project settings. 

#!/bin/bash
# start_workspace.sh: Domino pre-run script for Kiro
# Handles plugin discovery, project settings, and optional CLI install.
# Run once automatically at workspace start.

set -e

echo "=== Kiro Workspace Setup ==="

# ── 1. (Optional) Install Kiro CLI if not present ──
# Uncomment the following block to enable Kiro CLI in the workspace.
# if ! command -v kiro &> /dev/null; then
#     echo "[!] Kiro CLI not found. Installing..."
#     curl -fsSL https://cli.kiro.dev/install | bash
#     export PATH="$HOME/.local/bin:$PATH"
# fi
# echo "[ok] Kiro CLI version: $(kiro --version 2>/dev/null || echo 'installed')"

# ── 2. Register skills from the Domino plugin ──
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

mkdir -p /mnt/code/.kiro/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done

echo "[ok] Registered $(ls /mnt/code/.kiro/skills/ 2>/dev/null | wc -l) skills"

# ── 3. Populate domino_project_settings.md from environment ──
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Kiro setup complete ==="

What the script handles

  1. Optional CLI install: if the commented block is enabled and kiro is not on the PATH, installs it via the official installer.
  2. Register skills: symlinks all skills from the Domino plugin into the project's .kiro/skills/ directory so Kiro discovers them at session start.
  3. Populate project settings: writes $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME into domino_project_settings.md so the MCP server knows which project to operate against.

Template File Contents

.kiro/settings/mcp.json

Starts the Domino MCP server from the imported repo. Uses the mcpServers root key. Environment variables are injected automatically by Domino. Kiro supports environment variable expansion with the ${VARIABLE_NAME} syntax. 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

MCP configuration levels

LevelLocationScope
Workspace .kiro/settings/mcp.json This project only. Committed to the repository.
User ~/.kiro/settings/mcp.json All workspaces for the current user. Not committed.

MCP server properties

Each server entry in mcpServers supports these properties:

PropertyDescription
command + argsLocal server: the command and arguments to start the process.
url + headersRemote server: the endpoint URL and optional HTTP headers.
envEnvironment variables passed to the server. Supports ${VARIABLE_NAME} expansion.
disabledBoolean. Set to true to disable a server without removing its config.
autoApproveArray of tool names that do not require user confirmation before execution.
disabledToolsArray of tool names to disable from this server.

.kiro/steering/domino.md

The primary steering file for Domino platform rules. The inclusion: always frontmatter ensures it is loaded in every interaction. This is the Kiro equivalent of Claude Code's CLAUDE.md. Update it as your skills evolve and as the agent's purpose changes. 

---
inclusion: always
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md`
to get the `project_name` and `user_name`. If they contain placeholder
values, immediately update the file with the real values from
environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.kiro/settings/mcp.json`
and requires `DOMINO_API_KEY` and `DOMINO_HOST` environment variables.
Available tools: run_domino_job, check_domino_job_run_status,
check_domino_job_run_results, list_domino_project_files, file sync
utilities, get_domino_environment_info, open_web_browser.

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs
  rather than in the local terminal. This ensures governance,
  reproducibility, and audit trails.
- The Domino project name and user name are in
  domino_project_settings.md. Read this file before making any MCP
  tool calls.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- Commit and push before running jobs (Domino uses remote repo state).

## App Development

- Bind to 0.0.0.0 (not localhost) for Domino apps.
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

AGENTS.md

Kiro also reads AGENTS.md at the repository root. This file follows the standard format read by Claude Code and Codex, making it portable across all three agents. It is always loaded with no inclusion modes. You can use it as a simpler fallback or as a complement to the steering file. A typical approach: put the core Domino platform rules in both AGENTS.md and .kiro/steering/domino.md, then use additional steering files for Kiro-specific concerns like file-match patterns and auto-loaded context.

# Domino project settings to use with the MCP server domino_server

Steering Files

Steering files are Kiro's primary context system for persistent project guidance. They are Markdown files with YAML frontmatter placed in .kiro/steering/. Unlike skills, which load on demand, steering files can be configured to load automatically based on different conditions. They control platform behavior, coding standards, compliance rules, and anything else the agent should know about in every (or most) interactions.

Steering file locations

LevelLocationScope
Workspace .kiro/steering/ Project-specific. Committed to the repository. Takes priority over global.
Global ~/.kiro/steering/ All workspaces for the current user. Not committed.
Team ~/.kiro/steering/ (distributed via MDM) Pushed to all team members' machines via mobile device management.

When a workspace steering file and a global steering file have the same name, the workspace version takes priority. This allows teams to distribute default rules globally while letting individual projects override them.

Inclusion modes

The inclusion field in the YAML frontmatter determines when a steering file is loaded into the agent's context. There are four modes.

ModeFrontmatterWhen loaded
always inclusion: always Every interaction. This is the default for foundation files. Use for platform rules, compliance, and anything the agent must always know.
fileMatch inclusion: fileMatch + fileMatchPattern When the active file matches the glob pattern. Supports single patterns ("**/*.py") and arrays (["**/*.ts", "**/*.tsx"]).
manual inclusion: manual Only when explicitly referenced with #steering-file-name in chat or via / commands. Good for large reference documents.
auto inclusion: auto + name + description The agent decides whether to load the file based on the description matching the current request. Requires both name and description fields.

Example: always (platform rules)

---
inclusion: always
---

# Domino Platform Rules
All interactions must follow these rules...

Example: fileMatch (Python-specific guidance)

---
inclusion: fileMatch
fileMatchPattern: "**/*.py"
---

# Python Standards
When working with Python files, follow these conventions...

Example: fileMatch with multiple patterns

---
inclusion: fileMatch
fileMatchPattern:
  - "**/*.ts"
  - "**/*.tsx"
---

# TypeScript Standards
When working with TypeScript or TSX files...

Example: manual (large reference)

---
inclusion: manual
---

# API Reference
This document is loaded on demand with #api-reference in chat...

Example: auto (context-dependent)

---
inclusion: auto
name: deployment-checklist
description: Pre-deployment validation checklist for production releases.
---

# Deployment Checklist
The agent loads this when the request involves deployment...

Foundation steering files

Kiro supports three optional foundation steering files that are auto-generated and use inclusion: always by default. They provide broader project context that complements the Domino-specific domino.md file.

FilePurpose
product.mdProduct context: what the project does, who the users are, key workflows.
tech.mdTech stack: languages, frameworks, key dependencies, architecture decisions.
structure.mdProject structure: directory layout, module boundaries, naming conventions.

File references in steering files

Steering files can reference other project files using the #[[file:relative_path]] syntax. This lets you point the agent to specific configuration files, schemas, or examples without duplicating their content. 

See the MCP configuration at #[[file:.kiro/settings/mcp.json]]
and the project settings at #[[file:domino_project_settings.md]]

Security note. Never include API keys, tokens, or other secrets in steering files. They are committed to the repository. Use environment variables and the MCP server's env configuration instead.

Skills Format

Kiro supports Agent Skills, the same standard used by Claude Code and Codex. This means skills are fully portable between all three agents with no changes required. The Domino plugin ships 20 platform skills; your team adds domain-specific skills alongside them.

Steering files vs. skills

Kiro has two context systems that serve different purposes. Choosing the right one matters for performance and relevance.

SystemLocationLoadingBest for
Steering files .kiro/steering/ Based on inclusion mode (always, fileMatch, manual, auto) Platform behavior, coding standards, compliance rules that apply broadly.
Agent Skills .kiro/skills/ Progressive: metadata at startup, full content on demand Specific workflows, data access patterns, domain procedures relevant only sometimes.

Skill file locations

LevelLocationScope
Workspace .kiro/skills/<skill-name>/SKILL.md This project only. Committed to the repository or symlinked from the plugin.
Global ~/.kiro/skills/<skill-name>/SKILL.md All workspaces for the current user.

How skills load

  • Progressive loading: Kiro reads the metadata (name and description from the YAML frontmatter) at startup and loads the full skill content only when the agent determines it is relevant to the current request.
  • Explicit invocation: users can invoke any registered skill directly in chat by typing / to see available skills listed as slash commands, then selecting one.

SKILL.md anatomy

---
name: etl-pipeline
description: Run and monitor the nightly ETL pipeline on Domino.
---

# ETL Pipeline

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use.

## How to run
Step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes
Known failure modes and how to resolve them.

Frontmatter fields

FieldRequiredDescription
name Yes Must match the folder name. Lowercase letters, numbers, and hyphens only. Maximum 64 characters.
description Yes What the skill covers and when the agent should use it. Maximum 1024 characters. Write it as a specific, action-oriented sentence.
license No SPDX license identifier for the skill.
compatibility No Compatibility metadata for cross-agent portability.
metadata No Additional key-value pairs for custom metadata.

The description field is what Kiro uses to decide when to load the skill. A vague description means the skill may not load when needed. A good description is specific and action-oriented: "Run and monitor the nightly ETL pipeline on Domino" is better than "ETL stuff."

Hooks

Kiro supports event-driven automation through hooks. Hooks trigger on specific events and run predefined actions without manual intervention. They are useful for automating repetitive tasks like linting on save, pre-populating context, or running validation checks.

Supported event types

CategoryEvents
File events File created, file saved, file deleted
Prompt/Agent lifecycle Prompt submitted, response received, agent started, agent completed
Spec task events Spec task started, spec task completed
Manual triggers Triggered explicitly by the user through the UI or CLI

Hooks complement steering files and skills: steering files define what the agent knows, skills define what the agent can do, and hooks define what happens automatically in response to events.

Specs

Kiro's defining feature is spec-driven development. Specs are structured specifications that let you plan and build features methodically rather than through ad-hoc prompting. A spec breaks a feature into requirements, designs, tasks, and acceptance criteria before any code is written.

Specs are particularly useful in Domino workflows where reproducibility and auditability matter. By defining the full plan up front, you create a record of intent that complements Domino's job execution logs and experiment tracking.

Complete Template

Once the template is set up and the pre-run script has run, a fully built-out Kiro project looks like this:

/mnt/code/ ├── AGENTS.md ← System prompt (always loaded, portable) ├── domino_project_settings.md ← project_name and user_name └── .kiro/ ├── steering/ │ ├── domino.md ← Platform rules (always) │ ├── product.md ← Product context (optional, always) │ ├── tech.md ← Tech stack (optional, always) │ ├── structure.md ← Project structure (optional, always) │ ├── python-standards.md ← Python rules (fileMatch: **/*.py) │ └── deployment.md ← Deployment checklist (auto) ├── skills/ │ ├── [domino platform skills, symlinked from plugin] │ ├── etl-pipeline/SKILL.md ← Your workflow │ ├── data-sources/SKILL.md ← Your data access patterns │ ├── internal-apis/SKILL.md ← Your proprietary tools and APIs │ └── compliance/SKILL.md ← Your governance and regulatory rules └── settings/ └── mcp.json ← MCP server config

The Domino plugin skills are symlinked from /mnt/imported/code/domino-data-lab-plugin at workspace start. Your team's domain-specific skills are committed directly to the project repository under .kiro/skills/. Steering files are committed under .kiro/steering/.

Building Your Domain Skills

Out of the box, the template gives Kiro knowledge of the Domino platform only. The domain layer (your workflows, your data, your internal APIs) is empty. This is where your team's work begins. The Domino plugin is the model: a git repository of Markdown skill files, mounted as imported code and symlinked at workspace start. You follow the same pattern for your domain.

Where to put skills

There are two scopes:

  • Project-specific skills: add to .kiro/skills/ in your project git repo. They persist as long as you commit and push; a skill added in a workspace without pushing is lost when the workspace stops.
  • Org-wide skills: create a dedicated git repository (e.g. example-kiro-skills), attach it to the project template as Imported Code mounted at /mnt/imported/code/example-kiro-skills, and add one block to the pre-run script to symlink it alongside the Domino plugin:
# In start_workspace.sh, add after the existing plugin symlink loop
SKILLS_DIR="/mnt/imported/code/example-kiro-skills"

mkdir -p /mnt/code/.kiro/skills
for skill_dir in "$SKILLS_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done

Every project that mounts this repo picks up your org skills automatically on workspace start, the same pattern as the Domino plugin. Update the repo once; every project sees it.

Where to put steering files

There are also two scopes for steering files:

  • Project-specific: add to .kiro/steering/ in your project. Committed to the repository.
  • Global/team: place in ~/.kiro/steering/ on each team member's machine. Distribute via MDM for consistent team-wide rules. Workspace files override global files with the same name.

Extending

Custom MCP Servers

Write a custom MCP server when you need Kiro to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog. Register it by adding an entry to the mcpServers key in .kiro/settings/mcp.json alongside the Domino server: 

{
  "mcpServers": {
    "domino_server": { ... },
    "your_server": {
      "command": "python",
      "args": [".kiro/mcp-servers/your_server/your_server.py"]
    }
  }
}

Kiro also supports remote MCP servers. Use the url and headers properties instead of command and args: 

{
  "mcpServers": {
    "remote_server": {
      "url": "https://mcp.internal.example.com/sse",
      "headers": {
        "Authorization": "Bearer ${INTERNAL_API_TOKEN}"
      }
    }
  }
}

Agent Hooks

Use hooks to automate repetitive tasks tied to events. Examples: lint files on save, validate configuration when a YAML file is created, run a pre-commit check before the agent completes a task. Hooks reduce manual steps and enforce consistency across the team.

Explicit Skill Invocation

Invoke any registered skill directly in chat with /skill-name. This is useful for triggering a specific workflow without waiting for Kiro to auto-detect relevance. Type / in the chat to see the full list of available skills as slash commands.

Steering File References

Reference steering files manually in chat with #steering-file-name. This loads a manual inclusion steering file on demand. Useful for pulling in large reference documents only when needed.

Verification

To verify the setup is working:

  1. Create a new project from the template.
  2. Start a Domino workspace.
  3. Open the project in Kiro (IDE) or run kiro in the terminal (CLI).
  4. Ask: "What Domino skills do you have?"

Kiro should respond with the Domino platform skills from the plugin. If it does, the template is working correctly. When team members create new projects from the template, domino_project_settings.md is populated automatically at workspace start. They only need to add their domain skills to .kiro/skills/ and steering files to .kiro/steering/.
