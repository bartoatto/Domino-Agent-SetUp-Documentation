Domino Agentic Coding Setup Guide

A project template guide for configuring AI coding agents with Domino and your domain knowledge

Overview

AI coding agents can take actions on your behalf, running workflows, querying systems, analyzing data, and generating outputs, but out of the box they have no knowledge of Domino, your internal tooling, or your domain. This guide shows how to close that gap.

Why the Domino plugin matters

Without it, the agent has no knowledge of how Domino works. The plugin gives the agent everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why adding your own skills makes it more powerful

The plugin covers the platform. Your skills cover your domain. When you add skills for your data pipelines, internal APIs, and compliance rules, the agent stops being a generic assistant and starts understanding how your team specifically works.

Two layers

  • The Domino platform plugin, provided by Domino, covers the full platform. This is what you get out of the box.
  • Your skills, knowledge your team adds on top. Not included in the template. Explained in Part 4.

This guide covers the shared infrastructure that is the same regardless of which agent you use. For agent-specific configuration, see the individual guides:

Part 1: The Shared Domino Plugin

The plugin at /mnt/imported/code/domino-data-lab-plugin is sourced from dominodatalab/domino-claude-plugin. It provides 20 skills covering the full Domino platform: jobs, workspaces, environments, datasets, apps, model endpoints, MLflow, GenAI tracing, distributed computing, governance, and more.

It also bundles the domino_server MCP server, which gives the agent runtime tools:

  • run_domino_job, run a script as a Domino job
  • check_domino_job_run_status / check_domino_job_run_results, poll and retrieve job output
  • list_domino_project_files, browse project files
  • sync_local_file_to_domino / smart_sync_file / upload_file_to_domino_project / download_file_from_domino_project, file sync
  • get_domino_environment_info, inspect compute environments
  • open_web_browser, open Domino URLs

Agent-agnostic by design. The MCP server code is fully agent-agnostic, the same server works with every agent. Only the config file that points to it differs per agent.

Plugin directory structure

Part 2: Setting Up the Environment

Two things need to be created once and reused across all projects: a Domino environment and a Domino project template. The environment provides the runtime: system utilities, MCP server dependencies, and optionally the agent CLI. The template provides the project structure (covered in Part 3).

Common Dockerfile base

The following Dockerfile instructions are the same for all agents. They install the system utilities and MCP server dependencies that every agent needs.

# ── Install system utilities ──
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim iptables ipset && \
    rm -rf /var/lib/apt/lists/*

# ── Install MCP server dependencies (required) ──
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# ── Optional packages: add what your team needs ──
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Agent-specific additions (install commands, directories, environment variables) vary, see the individual agent guides for what to add on top of this base.

Base image requirements

The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement.

Note for IDE-based agents (Cursor, Kiro, Copilot): The agent itself does not need to be installed in the environment, it runs in the user's local IDE. Only the MCP server dependencies are needed in the Domino environment.

Part 3: Setting Up the Project Template

The template is a Domino project template backed by a git repository. It contains the configuration files that wire the agent to the Domino platform. An additional repository is attached as Imported Code; it mounts at a fixed path that the config files reference.

Shared across all agents: domino_project_settings.md

This file tells the MCP server which Domino project to operate against. It is auto-populated at workspace start by the pre-run script using environment variables. Users do not need to edit it.

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Imported Code (same for all agents)

Repository Mounts at
https://github.com/dominodatalab/domino-claude-plugin.git /mnt/imported/code/domino-data-lab-plugin

Agent-specific template files

Each agent requires its own system prompt file, MCP configuration file, and settings. These are covered in the individual agent guides. A single template can support multiple agents simultaneously, each agent reads only its own files, so they do not conflict.

Part 4: Building Your Domain Skills

Out of the box, the template gives the agent knowledge of the Domino platform only. The domain layer, your workflows, your data, your internal APIs, is empty. This is where your team's work begins.

Two scopes

  • Project-specific skills: live in your project git repo. They persist as long as you commit and push.
  • Org-wide skills: live in a dedicated git repository, attached to the project template as Imported Code, and symlinked at workspace start. Update the repo once; every project sees it.

The concept is the same regardless of agent, modular instruction files loaded on demand. The file format varies:

Agent Skills Directory File Format
Claude Code .claude/skills/*/ SKILL.md (Agent Skills standard)
Codex .agents/skills/*/ SKILL.md (Agent Skills standard)
Cursor .cursor/rules/ *.mdc with YAML frontmatter
Kiro .kiro/skills/*/ SKILL.md (Agent Skills standard)
GitHub Copilot .github/instructions/ *.instructions.md with applyTo

Portability: Claude Code, Codex, and Kiro all follow the open Agent Skills standard, skills are portable between them with minimal changes.

System prompt vs. Skills

  • System prompt = universal rules for every conversation (platform behavior, path conventions, compliance).
  • Skills = context-specific knowledge loaded on demand (a particular pipeline, a specific tool, a data access pattern).

Part 5: Custom MCP Servers

Most teams will not need this initially, skills cover a wide range of use cases. Write a custom MCP server when you need the agent to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, or fetching from a data catalog.

Example: minimal FastMCP server

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("server-1")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def tool_1(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Agent-agnostic. The MCP server code is fully agent-agnostic, the same Python server works with every agent. Only the config file that points to it differs.

Key principles

  • One tool per action, keep tools focused and composable.
  • Precise docstrings, the agent reads them to decide when to call the tool.
  • Return structured data, JSON objects, not free-form text.
  • Include clickable URLs, when your tool creates something with a UI, return the URL so the agent can share it with the user.
