Domino Claude Code Setup Guide

A project template guide for configuring Claude Code with Domino and your domain knowledge

Overview

Claude Code is Anthropic's agentic CLI that runs in the terminal. It reads CLAUDE.md for persistent instructions, supports MCP servers via .mcp.json, and loads skills from .claude/skills/ following the Agent Skills standard.

Out of the box, Claude Code has no knowledge of Domino Data Lab — how to run jobs, access datasets, deploy apps, or track experiments. The Domino plugin closes that gap. It bundles 20 platform skills and an MCP server that gives Claude Code runtime tools to take actions on the Domino platform on your behalf.

But the plugin only covers the platform. Your team's workflows, data pipelines, internal APIs, and compliance rules are a separate layer. By adding your own domain skills, you turn a generic assistant into one that understands how your organization specifically works.

Two layers, in short

Layer Source What it covers
Domino platform plugin Provided by Domino Jobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing, governance
Your domain skills Added by your team Your workflows, data pipelines, internal APIs, compliance rules

Part 1: How It Fits Together

The template ships with four configuration files in the project Git repository. Together they tell Claude Code what it is, how to behave, what tools it has, and how to connect to Domino.

Template file tree

/mnt/code/ ├── CLAUDE.md # System prompt (read every session) ├── domino_project_settings.md # Project name/user (read by MCP server) ├── .mcp.json # MCP server config └── .claude/ ├── settings.json # Plugin reference + permissions ├── skills/ # Symlinked from Domino plugin at workspace start ├── commands/ # Slash commands (symlinked from plugin) ├── agents/ # Specialized subagents (symlinked from plugin) └── output-styles/ # Output formatting (symlinked from plugin)

CLAUDE.md is read automatically at every session start. It contains the persistent system prompt — platform rules, data paths, compliance instructions. .claude/settings.json loads the Domino plugin (which registers skills, commands, agents, and output styles). .mcp.json starts the Domino MCP server and registers its runtime tools. domino_project_settings.md stores the current project and user name so MCP tools know which Domino project to target.

How skills arrive at runtime

The skills directories (.claude/skills/, .claude/commands/, etc.) are not checked into the project repo. They are populated at workspace start by the pre-run script, which symlinks them from the imported plugin repository. This means skills update automatically when the plugin is updated — no changes to the project repo required.

Plugin directory structure

The Domino plugin lives in an imported code repository mounted at /mnt/imported/code/domino-data-lab-plugin:

Part 2: The Shared Domino Plugin

The plugin lives at /mnt/imported/code/domino-data-lab-plugin, sourced from the dominodatalab/domino-claude-plugin repository. It provides two things: platform skills and an MCP server.

Platform skills (20 total)

Each skill is a directory containing a SKILL.md file following the Agent Skills standard. Skills are loaded on demand — only the name and description are read at startup; the full content loads when Claude determines it is relevant to the current conversation. The skills cover:

Category Skills
Compute jobs, workspaces, environments, distributed-computing, flows
Data datasets, data-connectivity, domino-data-sdk
Deployment app-deployment, model-endpoints, launchers, domino-ui-design
ML & AI experiment-tracking, modeling-assistant, model-monitoring, genai-tracing, ai-gateway
Platform projects, python-sdk, domino-governance

MCP server tools

The Domino MCP server (domino_mcp_server.py) provides runtime tools that Claude Code can call to take actions on the Domino platform:

Tool Purpose
run_domino_job Run a script as a Domino job
check_domino_job_run_status Poll job execution status
check_domino_job_run_results Retrieve job output/results
list_domino_project_files Browse project files on the platform
sync_local_file_to_domino Push a local file to the Domino project
smart_sync_file Intelligent file sync (handles conflicts)
upload_file_to_domino_project Upload a file to the project
download_file_from_domino_project Download a file from the project
get_domino_environment_info Inspect compute environments
open_web_browser Open Domino URLs in a browser

How the plugin loads

The plugin is activated through two mechanisms:

  • Plugin registration: The "plugins" key in .claude/settings.json points to the plugin directory. Claude Code reads the plugin manifest and registers all skills, commands, agents, and output styles.
  • MCP server startup: The .mcp.json file tells Claude Code to start the Domino MCP server process. Environment variables (DOMINO_USER_API_KEY, DOMINO_API_HOST) are injected automatically by Domino at workspace start.

Part 3: Setting Up the Claude-Domino Template and Environment

Part 3a: Setting Up the Environment

The workspace environment installs system utilities and MCP server dependencies at build time. Claude Code itself is either baked into the image (for air-gapped environments) or installed fresh at workspace start via the pre-run script (for environments with internet access).

Step 1: Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions.

Option A: Open internet — Claude Code installed at workspace start, always up to date:

USER root
RUN apt-get update && apt-get install -y ripgrep jq vim iptables ipset && rm -rf /var/lib/apt/lists/*
RUN mkdir -p /home/ubuntu/.claude /commandhistory && chown -R ubuntu:ubuntu /home/ubuntu/.claude /commandhistory
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"
RUN pip install --no-cache-dir fastmcp requests python-dotenv
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages streamlit-extras pandas pyarrow requests tenacity python-dateutil gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Option B: Air-gapped / restricted internet — Claude Code baked into the image. Add these lines after the system utilities and before the pip installs:

USER ubuntu
RUN curl -fsSL https://claude.ai/install.sh | bash
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"

Base image requirement: The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement.

Step 2: Pre-Run Script

The pre-run script runs automatically at every workspace start. It handles: installing Claude Code if not present, setting up per-user persistent storage, symlinking ~/.claude to durable storage, forcing fresh authentication (scrubbing OAuth credentials to prevent token leakage), persisting user preferences, registering plugin skills/commands/agents/output-styles, and populating domino_project_settings.md from environment variables. 

#!/bin/bash
# start_workspace.sh -- Domino pre-run script for AI coding agents
# Handles installation, per-user persistence, and plugin discovery.
# Run once automatically at workspace start.
#
# Security: OAuth credentials (~/.claude/.credentials.json) are scrubbed
# on each cold start to prevent token leakage via shared /mnt/artifacts.
# Preferences (~/.claude.json) are safe to persist -- they contain no secrets.

set -e

echo "=== Coding Agent Workspace Setup ==="

# ---------------------------------------------------------------
# 1. Install agent if not present
# ---------------------------------------------------------------
if ! command -v claude &> /dev/null; then
    echo "[!] Claude Code not found. Installing..."
    curl -fsSL https://claude.ai/install.sh | bash
    export PATH="$HOME/.local/bin:$PATH"
fi
echo "[ok] Claude Code version: $(claude --version 2>/dev/null || echo 'installed')"

# ---------------------------------------------------------------
# 2. Per-user persistent storage
# ---------------------------------------------------------------
if [ -n "$DOMINO_USER_NAME" ]; then
    PERSIST_USER="$DOMINO_USER_NAME"
else
    PERSIST_USER="$USER"
    echo "[!] DOMINO_USER_NAME not set, falling back to: $PERSIST_USER"
fi

PERSIST_DIR="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude"
mkdir -p "$PERSIST_DIR"
echo "[i] Persistent storage: $PERSIST_DIR"

# -- Symlink ~/.claude to persistent storage --
HOME_CLAUDE="$HOME/.claude"

if [ -d "$HOME_CLAUDE" ] && [ ! -L "$HOME_CLAUDE" ]; then
    echo "[i] Migrating existing ~/.claude to persistent storage..."
    cp -r "$HOME_CLAUDE/"* "$PERSIST_DIR/" 2>/dev/null || true
    rm -rf "$HOME_CLAUDE"
fi

[ -L "$HOME_CLAUDE" ] && [ ! -e "$HOME_CLAUDE" ] && rm -f "$HOME_CLAUDE"

if [ ! -L "$HOME_CLAUDE" ]; then
    ln -s "$PERSIST_DIR" "$HOME_CLAUDE"
    echo "[ok] ~/.claude symlinked to $PERSIST_DIR"
else
    echo "[ok] ~/.claude persistence already configured"
fi

# -- Force fresh authentication --
rm -f "$HOME/.claude/.credentials.json" 2>/dev/null || true
rm -f "$HOME/.claude/backups/.claude.json.backup."* 2>/dev/null || true
echo "[ok] Cleared OAuth credentials"

# -- Persist ~/.claude.json (preferences, not secrets) --
PERSIST_JSON="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude.json"
HOME_JSON="$HOME/.claude.json"

if [ ! -f "$PERSIST_JSON" ]; then
    if [ -f "$HOME_JSON" ] && [ ! -L "$HOME_JSON" ]; then
        cp "$HOME_JSON" "$PERSIST_JSON"
    else
        echo '{}' > "$PERSIST_JSON"
    fi
fi

rm -f "$HOME_JSON" 2>/dev/null || true

if [ ! -L "$HOME_JSON" ]; then
    ln -s "$PERSIST_JSON" "$HOME_JSON"
    echo "[ok] ~/.claude.json symlinked to $PERSIST_JSON"
else
    echo "[ok] ~/.claude.json persistence already configured"
fi

# ---------------------------------------------------------------
# 3. Register Domino plugin (skills, commands, agents)
# ---------------------------------------------------------------
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

# -- Skills (SKILL.md format, Agent Skills standard) --
mkdir -p /mnt/code/.claude/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.claude/skills/$skill_name" 2>/dev/null || true
done

# -- Commands --
mkdir -p /mnt/code/.claude/commands
for cmd_file in "$PLUGIN_DIR"/commands/*.md; do
    [ -f "$cmd_file" ] || continue
    ln -sf "$cmd_file" "/mnt/code/.claude/commands/$(basename "$cmd_file")" 2>/dev/null || true
done

# -- Agents --
mkdir -p /mnt/code/.claude/agents
for agent_file in "$PLUGIN_DIR"/agents/*.md; do
    [ -f "$agent_file" ] || continue
    ln -sf "$agent_file" "/mnt/code/.claude/agents/$(basename "$agent_file")" 2>/dev/null || true
done

# -- Output styles --
mkdir -p /mnt/code/.claude/output-styles
for style_file in "$PLUGIN_DIR"/output-styles/*.md; do
    [ -f "$style_file" ] || continue
    ln -sf "$style_file" \
      "/mnt/code/.claude/output-styles/$(basename "$style_file")" 2>/dev/null || true
done

echo "[ok] Registered skills: $(ls /mnt/code/.claude/skills/ 2>/dev/null | wc -l)"

# ---------------------------------------------------------------
# 4. Populate domino_project_settings.md from environment
# ---------------------------------------------------------------
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

echo "=== Coding agent setup complete ==="

Part 3b: Setting Up the Template

The template is a Domino project template backed by a Git repository. When a team member creates a new project from the template, they get the Claude Code configuration files pre-loaded.

Step 1: Create the Git Repository

Create a new Git repository containing these four files:

.claude/settings.json

Loads the Domino plugin and grants the permissions Claude Code needs:

{
  "plugins": [
    "/mnt/imported/code/domino-data-lab-plugin"
  ],
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(python *)",
      "Bash(pip *)",
      "Bash(claude *)"
    ]
  }
}

.mcp.json

Starts the Domino MCP server. Environment variables are injected automatically by Domino:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. The placeholder values are replaced with real environment variables:

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

CLAUDE.md

The persistent system prompt, read at every session start:

# Project Instructions for Claude Code

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.mcp.json` and requires
`DOMINO_API_KEY` and `DOMINO_HOST` environment variables. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve job output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` / `download_file_from_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

Part 4: Building Your Domain Skills

The Domino plugin gives Claude Code knowledge of the platform. Your domain skills give it knowledge of your workflows, data, and internal systems.

Two scopes for skills

Scope Location How it works
Project-specific .claude/skills/ in your project repo Committed directly to the project. Available only in that project.
Org-wide A separate Git repo mounted as Imported Code Symlinked by the pre-run script, available to every project that imports the repo.

Skill anatomy

Each skill is a directory containing a SKILL.md file with YAML frontmatter:

my-skill/ SKILL.md # Required: instructions + frontmatter scripts/ # Optional: executable code references/ # Optional: documentation assets/ # Optional: templates

SKILL.md format:

---
name: my-skill
description: What this skill covers and when the agent should use it.
---

# My Skill

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes

The description field is critical — it determines when the agent activates the skill. Write it as a specific, action-oriented sentence. For example: "Deploy and monitor model API endpoints in Domino. Use when deploying models as APIs or monitoring endpoint health."

CLAUDE.md vs. Skills

CLAUDE.md (System Prompt) Skills (.claude/skills/)
When loaded Every session, automatically On demand, when Claude determines it is relevant
What belongs here Universal rules: platform behavior, path conventions, compliance, identity Context-specific knowledge for a particular workflow or tool
Size guideline Keep concise — loaded every time Can be detailed — only loaded when needed

If a rule applies to every conversation (e.g., "always commit before running jobs"), it belongs in CLAUDE.md. If it only applies when running a specific pipeline or using a specific tool, it belongs in a skill.

Part 6: Extending Further

Slash commands

For repetitive multi-step workflows, add slash commands to .claude/commands/. Each command is a Markdown file. Users type /command-name and Claude expands it to a full set of instructions.

Good candidates for slash commands:

  • Initializing new jobs from templates
  • Scaffolding experiments with standard MLflow setup
  • Generating data access request templates
  • Running standard validation/testing workflows

Example: .claude/commands/run-training.md

Run the training script as a Domino job:
1. Check for uncommitted changes and commit/push if needed
2. Run `scripts/train.py` as a Domino job
3. Monitor the job status until completion
4. Report the results and any MLflow experiment URLs

Custom MCP servers

Most teams will not need this initially — skills cover a wide range of use cases. Write a custom MCP server when you need Claude Code to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

A minimal MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("server-1")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def tool_1(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Register your custom MCP server in .mcp.json alongside the Domino server: 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "my_custom_server": {
      "command": "python",
      "args": ["/mnt/code/mcp-servers/my_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}"
      }
    }
  }
}

Key principles: one tool per action, precise docstrings (Claude reads them to decide when to call the tool), return structured data, and include clickable URLs in responses when your tool creates something with a UI.
