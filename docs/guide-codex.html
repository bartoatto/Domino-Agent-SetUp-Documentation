Domino Codex Setup Guide

A project template guide for configuring OpenAI Codex with Domino and your domain knowledge

Overview

OpenAI Codex is OpenAI's agentic coding CLI that runs directly in the terminal. It reads AGENTS.md for persistent instructions, supports MCP (Model Context Protocol) servers via .codex/config.toml, and loads skills from .agents/skills/ following the open Agent Skills standard. Install it with npm install -g @openai/codex (requires Node.js 22+), then run codex in any project directory.

Out of the box, Codex has no knowledge of the Domino Data Lab platform. It does not know how to run Domino Jobs, access Domino Datasets, deploy Domino Apps, or track experiments with MLflow. The Domino plugin closes that gap entirely: it provides an MCP server that gives Codex runtime tools for interacting with the Domino API, plus 20 domain skills covering jobs, workspaces, environments, datasets, apps, model endpoints, MLflow experiment tracking, GenAI tracing, distributed computing, governance, and more.

But platform knowledge alone is not enough. Your organization has its own data pipelines, internal APIs, compliance rules, and domain-specific workflows. By adding your own skills alongside the Domino plugin, you transform Codex from a generic assistant into one that understands how your team specifically works.

Two layers

Layer Source What it covers
Domino platform plugin Provided by Domino Jobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing, governance
Your domain skills Added by your team Your workflows, data pipelines, internal APIs, compliance rules

Part 1: How It Fits Together

The Codex-Domino project template ships with a small set of configuration files. Each file has a specific role in wiring Codex to the Domino platform:

/mnt/code/ ├── AGENTS.md ← Domino-specific rules (system prompt) ├── domino_project_settings.md ← Project name/user (read by MCP server) └── .codex/ └── config.toml ← MCP servers, sandbox, permissions

AGENTS.md is the system prompt. Codex reads it automatically at every session start. It tells Codex what it is (a Domino-powered agentic tool), what MCP tools are available, how to behave (run jobs instead of local commands, commit before running, check job results), and platform-specific conventions (data paths, app binding rules).

domino_project_settings.md stores the current project name and username. The MCP server reads this file to know which Domino project to operate against. It is auto-populated at workspace start by the pre-run script.

.codex/config.toml is the Codex configuration file in TOML format. It declares MCP servers (including the Domino MCP server), sandbox settings, and approval policies.

Skills: delivered at workspace start

Skills are not stored in the template repository itself. They live in the shared Domino plugin repository, which is attached as Imported Code and mounted at /mnt/imported/code/domino-data-lab-plugin/. A pre-run script symlinks the plugin's skills into .agents/skills/ (the directory Codex discovers skills from) every time a workspace starts.

The plugin repository has the following structure:

/mnt/imported/code/domino-data-lab-plugin/ ├── skills/ ← 20 Domino platform skills (SKILL.md format) │ ├── ai-gateway/ │ ├── app-deployment/ │ ├── data-connectivity/ │ ├── datasets/ │ ├── distributed-computing/ │ ├── domino-data-sdk/ │ ├── domino-governance/ │ ├── domino-ui-design/ │ ├── environments/ │ ├── experiment-tracking/ │ ├── flows/ │ ├── genai-tracing/ │ ├── jobs/ │ ├── launchers/ │ ├── model-endpoints/ │ ├── model-monitoring/ │ ├── modeling-assistant/ │ ├── projects/ │ ├── python-sdk/ │ └── workspaces/ ├── mcp-servers/ │ └── domino_mcp_server/ │ └── domino_mcp_server.py ← The Domino MCP server (shared across agents) ├── agents/ ← Specialized subagents (Claude Code specific) ├── commands/ ← Slash commands (Claude Code specific) ├── output-styles/ ← Output formatting (Claude Code specific) ├── hooks/ ← Pre/post hooks ├── templates/ ← Project template files └── .claude-plugin/ ← Claude Code plugin manifest

The key insight: the MCP server and skills are agent-agnostic. The same domino_mcp_server.py serves tools to Claude Code, Codex, Cursor, Kiro, and Copilot. The same SKILL.md files work with Claude Code, Codex, and Kiro (all follow the Agent Skills standard). Only the configuration file that points to them differs.

MCP server tools

The Domino MCP server (domino_mcp_server.py) exposes runtime tools that Codex can call during a session. These are configured in .codex/config.toml:

  • run_domino_job — run a script as a Domino Job
  • check_domino_job_run_status — poll a running job for its current state
  • check_domino_job_run_results — retrieve stdout/stderr from a completed job
  • list_domino_project_files — browse files in the Domino project
  • sync_local_file_to_domino / smart_sync_file / upload_file_to_domino_project / download_file_from_domino_project — file synchronization
  • get_domino_environment_info — inspect compute environments
  • open_web_browser — open Domino URLs

Skill discovery path

Codex discovers skills from .agents/skills/ in the project directory. The pre-run script symlinks each skill directory from the plugin into this path at workspace start. No additional configuration is needed in config.toml — Codex automatically scans the directory, reads each SKILL.md frontmatter at startup, and loads the full content on demand when it determines a skill is relevant to the current task.

Part 3: Setting Up the Codex-Domino Template and Environment

Part 3a: Environment

Codex requires Node.js 22+ to install and run. The Domino compute environment must provide Node.js at this version (or newer), along with system utilities and MCP server dependencies.

Step 1: Dockerfile instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions.

Option A: Open internet — Codex installed at workspace start, always up to date:

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*

# -- Install Node.js 22+ if not already present --
RUN curl -fsSL https://deb.nodesource.com/setup_22.x | bash - && \
    apt-get install -y nodejs && \
    rm -rf /var/lib/apt/lists/*

# -- Install MCP server dependencies (required) --
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional -- add what your team needs) --
RUN pip install dash streamlit pandas pyarrow
USER ubuntu

Option B: Air-gapped / restricted internet — Codex baked into the image:

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*

# -- Install Node.js 22+ if not already present --
RUN curl -fsSL https://deb.nodesource.com/setup_22.x | bash - && \
    apt-get install -y nodejs && \
    rm -rf /var/lib/apt/lists/*

# -- Install Codex into the image --
USER ubuntu
RUN npm install -g @openai/codex
ENV PATH="/home/ubuntu/.local/bin:$PATH"

# -- Install MCP server dependencies (required) --
USER root
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional) --
RUN pip install dash streamlit pandas pyarrow
USER ubuntu

Base image requirement: The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement.

Step 2: Pre-run script

The pre-run script runs automatically at every workspace start. It handles: installing Codex if not already present, symlinking plugin skills into .agents/skills/, and populating domino_project_settings.md with environment variables.

Unlike Claude Code, Codex does not have a ~/.claude persistence model with OAuth credentials and preference files. Authentication is handled via the OPENAI_API_KEY environment variable (configured as a Domino user secret) or ChatGPT login. The pre-run script is therefore simpler: 

#!/bin/bash
# start_workspace.sh -- Domino pre-run script for Codex
# Handles installation, skill discovery, and project settings.

set -e

echo "=== Codex Workspace Setup ==="

# ---------------------------------------------------------------
# 1. Install Codex if not present (requires Node.js 22+)
# ---------------------------------------------------------------
if ! command -v codex &> /dev/null; then
    echo "[!] Codex not found. Installing..."
    npm install -g @openai/codex
fi
echo "[ok] Codex version: $(codex --version 2>/dev/null || echo 'installed')"

# ---------------------------------------------------------------
# 2. Symlink Domino plugin skills to .agents/skills/
# ---------------------------------------------------------------
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

mkdir -p /mnt/code/.agents/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

echo "[ok] Registered skills: $(ls /mnt/code/.agents/skills/ 2>/dev/null | wc -l)"

# ---------------------------------------------------------------
# 3. Populate domino_project_settings.md from environment
# ---------------------------------------------------------------
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

echo "=== Codex setup complete ==="

Part 3b: Template

The template is a Domino project template backed by a Git repository. When a team member creates a new project from the template, they get the Codex configuration files pre-loaded.

Step 1: Create the Git repository

Create a new Git repository containing these files:

/mnt/code/ ├── AGENTS.md ← System prompt ├── domino_project_settings.md ← Project name/user (auto-populated) └── .codex/ └── config.toml ← MCP server, sandbox, approval policy

Step 2: File contents

AGENTS.md — the system prompt read at every Codex session:

# Project Instructions for Codex

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.codex/config.toml`. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve job output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` / `download_file_from_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

.codex/config.toml — Codex configuration in TOML format:

# ── MCP Servers ──────────────────────────────────────────────

[mcp_servers.domino_server]
command = "python"
args = ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"]

[mcp_servers.domino_server.env]
DOMINO_API_KEY = "${DOMINO_USER_API_KEY}"
DOMINO_HOST = "${DOMINO_API_HOST}"

# ── Sandbox & Approval ──────────────────────────────────────

# Options: "read-only", "workspace-write", "danger-full-access"
sandbox_mode = "workspace-write"

# Options: "untrusted" (ask for everything), "on-request", "never"
approval_policy = "on-request"

# ── Skills ───────────────────────────────────────────────────

# Codex discovers skills automatically from .agents/skills/
# No additional configuration needed.

domino_project_settings.md — auto-populated at workspace start:

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Part 4: Building Your Domain Skills

The plugin covers the Domino platform. Your skills cover your domain. This section explains how to write skills that teach Codex about your organization's specific workflows, data pipelines, internal APIs, and compliance rules.

The Agent Skills standard

Codex follows the open Agent Skills standard, the same format used by Claude Code and Kiro. A skill written once in this format works across all three agents. Each skill is a directory containing a SKILL.md file:

my-skill/ ├── SKILL.md ← Required: instructions + frontmatter ├── scripts/ ← Optional: executable code ├── references/ ← Optional: documentation └── assets/ ← Optional: templates

SKILL.md format:

---
name: my-skill
description: What this skill covers and when the agent should use it.
---

# My Skill

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes

The description field is critical — it determines when Codex activates the skill. Write it as a specific, action-oriented sentence. Codex reads only the frontmatter at startup and loads the full body on demand when it determines the skill is relevant.

Skill discovery paths

Codex discovers skills from three locations, in order:

Location Scope Use case
.agents/skills/ (in project repo) Project-specific Skills that apply to this project only
~/.agents/skills/ (user home) Per-user Skills a user carries across all projects
/etc/codex/skills/ (system-wide) Admin/org-wide Skills baked into the environment image

Project-specific vs. org-wide skills

Project-specific skills: Add directly to .agents/skills/ in your project Git repo. Commit and push; skills persist as long as they are in the repo.

Org-wide skills: Create a dedicated Git repository (e.g., my-org-skills), attach it to the project template as Imported Code, and add a block to the pre-run script to symlink it alongside the Domino plugin: 

# In start_workspace.sh, add after the existing plugin symlink loop
ORG_SKILLS="/mnt/imported/code/my-org-skills"

for skill_dir in "$ORG_SKILLS"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

Every project that mounts this repo picks up your org skills automatically on workspace start.

AGENTS.md vs. Skills

AGENTS.md (the system prompt): universal rules for every conversation — platform behavior, path conventions, compliance defaults, MCP tool inventory. This is loaded at every session, regardless of what the user asks about.

Skills: context-specific knowledge for a particular workflow. If a rule only applies when running a specific pipeline or tool, it belongs in a skill. Skills are loaded on demand, keeping the context window efficient.

Part 5: The Complete Template

Here is the full file tree of a Codex-Domino project template with both platform skills (from the plugin) and custom domain skills:

/mnt/code/ ├── AGENTS.md ← System prompt (Codex reads at every session) ├── domino_project_settings.md ← Project name/user (auto-populated) ├── .codex/ │ └── config.toml ← MCP servers, sandbox, approval policy └── .agents/ └── skills/ ← Discovered by Codex automatically ├── ai-gateway/ ┐ ├── app-deployment/ │ ├── data-connectivity/ │ ├── datasets/ │ ├── distributed-computing/ │ ├── domino-data-sdk/ │ ├── domino-governance/ │ Domino platform skills ├── domino-ui-design/ │ (symlinked from plugin at ├── environments/ │ workspace start) ├── experiment-tracking/ │ ├── flows/ │ ├── genai-tracing/ │ ├── jobs/ │ ├── launchers/ │ ├── model-endpoints/ │ ├── model-monitoring/ │ ├── modeling-assistant/ │ ├── projects/ │ ├── python-sdk/ │ ├── workspaces/ ┘ ├── my-data-pipeline/ ┐ ├── my-internal-api/ │ Your custom domain skills └── my-compliance-rules/ ┘ (committed to project repo)

The platform skills are symlinks — they point to the mounted plugin repository and stay up to date whenever the Imported Code reference is updated. Your custom skills are regular directories committed to your project's Git repo.

At session start, Codex scans .agents/skills/, reads the SKILL.md frontmatter from each directory, and registers them for on-demand loading. When you ask Codex to do something that matches a skill's description, it loads the full content and follows the instructions.

Part 6: Extending Further

Custom MCP servers

Most teams will not need this initially; skills cover a wide range of use cases. Write a custom MCP server when you need Codex to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

A minimal MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("my-internal-server")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def query_internal_system(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

The MCP server code is fully agent-agnostic. To register it with Codex, add a new section to .codex/config.toml (instead of .mcp.json as you would for Claude Code): 

# Add to .codex/config.toml alongside the existing domino_server

[mcp_servers.my_internal_server]
command = "python"
args = ["/mnt/code/mcp-servers/my_internal_server.py"]

[mcp_servers.my_internal_server.env]
MY_API_TOKEN = "${MY_API_TOKEN}"

Key principles: one tool per action, precise docstrings (Codex reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Profiles

Codex supports named configuration profiles for different use cases. Define them in config.toml: 

[profiles.fast]
model = "gpt-5-codex"
service_tier = "fast"

[profiles.research]
model = "gpt-5-codex"
model_reasoning_effort = "xhigh"

Switch at launch with:

codex --profile fast

This is useful for giving team members a "fast" mode for quick tasks and a "research" mode for complex analysis that benefits from deeper reasoning.

Multi-agent support

Codex supports spawning sub-agents for parallel work. Enable it in config.toml: 

[features]
multi_agent = true

[agents]
max_threads = 6
max_depth = 1

With multi-agent enabled, Codex can delegate subtasks to parallel threads — for example, running data validation on multiple datasets simultaneously or generating code for several components at once.

Network access control

Control outbound network access per sandbox mode. This is useful for restricting Codex to only communicate with your Domino instance and approved external services: 

[permissions.default.network]
enabled = true
allowed_domains = ["*.domino.tech", "github.com"]

Context loading: how it works

Understanding how Codex assembles its context helps you write better AGENTS.md files and organize skills effectively:

  1. Global instructions: ~/.codex/AGENTS.md (or AGENTS.override.md) loads first
  2. Project instructions: Codex walks from the repo root to the current working directory, concatenating AGENTS.md files at each level
  3. Override precedence: AGENTS.override.md takes priority over AGENTS.md at any level
  4. Size limit: Combined instructions capped at 32 KiB (project_doc_max_bytes)
  5. MCP servers: Started from config.toml; tools registered at session start
  6. Skills: Discovered from .agents/skills/*/SKILL.md — metadata at startup, full content on demand
