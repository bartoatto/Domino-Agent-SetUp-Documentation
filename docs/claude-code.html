Claude Code on Domino

Agent-specific configuration for Claude Code

Claude Code is Anthropic's agentic CLI that runs in the terminal. It can take actions on your behalf: running workflows, querying systems, analyzing data, and generating outputs. Out of the box, it has no knowledge of Domino, your internal tooling, or your domain. This page shows how to close that gap.

Note: The shared setup, the Domino plugin, common Dockerfile dependencies, and the project template concept, is covered on the Overview page. This page focuses on the Claude Code-specific configuration.

Template Files

The project template ships with four files that wire Claude Code to the Domino platform:

/mnt/code/ ├── CLAUDE.md ← System prompt (read every session) ├── domino_project_settings.md ← Project name/user (shared) ├── .mcp.json ← MCP server config └── .claude/ └── settings.json ← Plugin reference + permissions
FilePurpose
CLAUDE.md Read automatically at the start of every Claude Code session. Tells Claude what it is, what it can do, and how to behave. Update it as your skills evolve.
.claude/settings.json Loads the Domino plugin via the "plugins" key and grants permissions Claude needs (git, python, pip, claude).
.mcp.json Starts the Domino MCP server at workspace launch. Provides runtime tools like run_domino_job, check_domino_job_run_status, file sync utilities, and more.
domino_project_settings.md Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script.

Environment Setup

Claude Code needs to be available in the workspace. There are two options depending on network access.

Option A: Open Internet (Pre-Run Script)

Claude Code is installed fresh at every workspace start, meaning it is always up to date. No image rebuild needed.

curl -fsSL https://claude.ai/install.sh | bash

This is handled automatically by the pre-run script (see below). The Dockerfile only needs the supporting directories and dependencies.

Option B: Air-Gapped (Dockerfile)

Claude Code is baked into the image at build time. To update, rebuild the image.

# Install Claude Code into the image
USER ubuntu
RUN curl -fsSL https://claude.ai/install.sh | bash
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"

Required Directories (Both Options)

The Dockerfile must create the directories Claude Code expects:

# Create directories for Claude Code
RUN mkdir -p /home/ubuntu/.claude /commandhistory && \
    chown -R ubuntu:ubuntu /home/ubuntu/.claude /commandhistory
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"

MCP Server Dependencies

The MCP server requires these Python packages (install in the Dockerfile):

RUN pip install --no-cache-dir fastmcp requests python-dotenv

Pre-Run Script

The pre-run script runs automatically at every workspace start. It handles installation, per-user persistent storage, plugin discovery, and project settings. Add this as the workspace pre-run script in your Domino environment or project settings. 

#!/bin/bash
# start_workspace.sh: Domino pre-run script for Claude Code
# Handles installation, per-user persistence, and plugin discovery.
# Run once automatically at workspace start.
#
# Security: OAuth credentials (~/.claude/.credentials.json) are scrubbed
# on each cold start to prevent token leakage via shared /mnt/artifacts.
# Preferences (~/.claude.json) are safe to persist: they contain no secrets.

set -e

echo "=== Claude Code Workspace Setup ==="

# ── 1. Install Claude Code if not present ──
if ! command -v claude &> /dev/null; then
    echo "[!] Claude Code not found. Installing via native installer..."
    curl -fsSL https://claude.ai/install.sh | bash
    export PATH="$HOME/.local/bin:$PATH"
fi

echo "[✓] Claude Code version: $(claude --version 2>/dev/null || echo 'installed')"

# ── 2. Determine per-user persistent storage ──
if [ -n "$DOMINO_USER_NAME" ]; then
    PERSIST_USER="$DOMINO_USER_NAME"
else
    PERSIST_USER="$USER"
    echo "[!] DOMINO_USER_NAME not set, falling back to: $PERSIST_USER"
fi

PERSIST_DIR="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude"
mkdir -p "$PERSIST_DIR"
echo "[i] Persistent storage: $PERSIST_DIR"

# ── 3. Symlink ~/.claude to persistent storage ──
HOME_CLAUDE="$HOME/.claude"

if [ -d "$HOME_CLAUDE" ] && [ ! -L "$HOME_CLAUDE" ]; then
    echo "[i] Migrating existing ~/.claude to persistent storage..."
    cp -r "$HOME_CLAUDE/"* "$PERSIST_DIR/" 2>/dev/null || true
    rm -rf "$HOME_CLAUDE"
fi

[ -L "$HOME_CLAUDE" ] && [ ! -e "$HOME_CLAUDE" ] && rm -f "$HOME_CLAUDE"

if [ ! -L "$HOME_CLAUDE" ]; then
    ln -s "$PERSIST_DIR" "$HOME_CLAUDE"
    echo "[✓] ~/.claude symlinked to $PERSIST_DIR"
else
    echo "[✓] ~/.claude persistence already configured"
fi

# ── 4. Force fresh authentication ──
rm -f "$HOME/.claude/.credentials.json" 2>/dev/null || true
rm -f "$HOME/.claude/backups/.claude.json.backup."* 2>/dev/null || true
echo "[✓] Cleared OAuth credentials: you will need to re-authenticate"

# ── 5. Persist ~/.claude.json (preferences, not secrets) ──
PERSIST_JSON="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude.json"
HOME_JSON="$HOME/.claude.json"

if [ ! -f "$PERSIST_JSON" ]; then
    if [ -f "$HOME_JSON" ] && [ ! -L "$HOME_JSON" ]; then
        cp "$HOME_JSON" "$PERSIST_JSON"
    else
        echo '{}' > "$PERSIST_JSON"
    fi
fi

rm -f "$HOME_JSON" 2>/dev/null || true

if [ ! -L "$HOME_JSON" ]; then
    ln -s "$PERSIST_JSON" "$HOME_JSON"
    echo "[✓] ~/.claude.json symlinked to $PERSIST_JSON"
else
    echo "[✓] ~/.claude.json persistence already configured"
fi

# ── 6. Register plugins (skills, commands, agents, output-styles) ──
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

mkdir -p /mnt/code/.claude/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.claude/skills/$skill_name" 2>/dev/null || true
done

mkdir -p /mnt/code/.claude/commands
for cmd_file in "$PLUGIN_DIR"/commands/*.md; do
    [ -f "$cmd_file" ] || continue
    ln -sf "$cmd_file" "/mnt/code/.claude/commands/$(basename "$cmd_file")" 2>/dev/null || true
done

mkdir -p /mnt/code/.claude/agents
for agent_file in "$PLUGIN_DIR"/agents/*.md; do
    [ -f "$agent_file" ] || continue
    ln -sf "$agent_file" "/mnt/code/.claude/agents/$(basename "$agent_file")" 2>/dev/null || true
done

mkdir -p /mnt/code/.claude/output-styles
for style_file in "$PLUGIN_DIR"/output-styles/*.md; do
    [ -f "$style_file" ] || continue
    ln -sf "$style_file" \
      "/mnt/code/.claude/output-styles/$(basename "$style_file")" 2>/dev/null || true
done

echo "[✓] Registered $(ls /mnt/code/.claude/skills/ 2>/dev/null | wc -l) skills, \
$(ls /mnt/code/.claude/commands/ 2>/dev/null | wc -l) commands, \
$(ls /mnt/code/.claude/agents/ 2>/dev/null | wc -l) agents"

# ── 7. Populate domino_project_settings.md from environment ──
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[✓] domino_project_settings.md populated from environment"
fi

echo "=== Claude Code setup complete ==="

What the script handles

  1. Install Claude Code, if claude is not on the PATH, installs it via the official installer.
  2. Per-user persistent storage, symlinks ~/.claude to /mnt/artifacts/.claude-users/$USER/.claude so settings survive workspace restarts.
  3. Force fresh authentication, scrubs OAuth credentials (.credentials.json) on each cold start to prevent token leakage via shared artifacts.
  4. Persist preferences, symlinks ~/.claude.json to persistent storage. Preferences contain no secrets and are safe to persist.
  5. Register plugins, symlinks skills, commands, agents, and output-styles from the Domino plugin into the project's .claude/ directory.
  6. Populate project settings, writes $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME into domino_project_settings.md.

Template File Contents

.claude/settings.json

Loads the Domino plugin and grants the permissions Claude needs:

{
  "plugins": [
    "/mnt/imported/code/domino-data-lab-plugin"
  ],
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(python *)",
      "Bash(pip *)",
      "Bash(claude *)"
    ]
  }
}

.mcp.json

Starts the Domino MCP server from the imported repo. Environment variables are injected automatically by Domino:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. Users do not need to edit this file.

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

CLAUDE.md

The persistent system prompt for Claude Code, read automatically at every session. Start with the core Domino platform rules below and extend with your organization's requirements.

# Project Instructions for Claude Code

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.


## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to
get the `project_name` and `user_name`. If they contain placeholder
values, **immediately update the file** with the real values from
environment variables (do not wait to be asked):
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

This must also happen automatically at `/init` time.


## MCP Server

The `domino_server` MCP is configured in `.mcp.json` and requires
`DOMINO_API_KEY` and `DOMINO_HOST` environment variables. Available tools:
- `run_domino_job`: run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results`: poll
  and retrieve job output
- `list_domino_project_files`: browse project files
- `sync_local_file_to_domino` / `smart_sync_file` /
  `upload_file_to_domino_project` /
  `download_file_from_domino_project`: file sync
- `get_domino_environment_info`: inspect compute environments
- `open_web_browser`: open Domino URLs


## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs
  rather than in the local terminal. This ensures governance,
  reproducibility, and audit trails.
- The Domino project name and user name are in
  domino_project_settings.md. Read this file before making any MCP
  tool calls.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.


## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.


## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.


## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

Skills Format

Claude Code uses the Agent Skills standard. Skills are Markdown files placed in .claude/skills/<skill-name>/SKILL.md. The Domino plugin ships 20 platform skills; your team adds domain-specific skills alongside them.

SKILL.md Anatomy

---
name: skill-name
description: What this skill covers and when Claude should use it.
---

# Skill Name

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages Claude should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes
Known failure modes and how to resolve them.

The description field in the frontmatter is what Claude uses to decide when to load the skill. Write it as a specific, action-oriented sentence. Skills load only when relevant.

CLAUDE.md vs. Skills

FileScopeWhen to use
CLAUDE.md Universal Rules for every conversation: platform behavior, path conventions, compliance.
SKILL.md Context-specific Knowledge for a particular workflow. If a rule only applies when running a specific pipeline or tool, it belongs in a skill.

Complete Template

Once the template is set up and the pre-run script has run, a fully built-out project looks like this:

/mnt/code/ ├── CLAUDE.md ← Platform rules, data paths, compliance ├── domino_project_settings.md ← project_name and user_name ├── .mcp.json ← MCP server config └── .claude/ ├── settings.json ← References shared Domino plugin ├── agents/ ← Symlinked from Domino plugin ├── commands/ ← Symlinked from Domino plugin ├── output-styles/ ← Symlinked from Domino plugin └── skills/ ├── [domino platform skills, symlinked from plugin] ├── etl-pipeline/SKILL.md ← Your workflow ├── data-sources/SKILL.md ← Your data access patterns └── compliance/SKILL.md ← Your governance rules

The Domino plugin skills, commands, agents, and output-styles are symlinked from /mnt/imported/code/domino-data-lab-plugin at workspace start. Your team's domain-specific skills are committed directly to the project repository.

Extending

Slash Commands

For repetitive multi-step workflows, add slash commands to .claude/commands/. Users type /command-name and Claude expands it to a full set of instructions. Good candidates: initializing new jobs from templates, scaffolding experiments with standard MLflow setup, generating data access request templates.

Custom MCP Servers

Write a custom MCP server when you need Claude to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog. Register it by adding an entry to the mcpServers key in .mcp.json: 

{
  "mcpServers": {
    "domino_server": { ... },
    "your_server": {
      "command": "python",
      "args": [".claude/mcp-servers/your_server/your_server.py"]
    }
  }
}

Verification

To verify the setup is working:

  1. Create a new project from the template.
  2. Start an IDE workspace and open a terminal.
  3. Run claude in the terminal.
  4. Ask: "What Domino skills do you have?"

Claude should respond with the Domino platform skills from the plugin. If it does, the template is working correctly. Team members who create new projects from the template only need to add their domain skills to .claude/skills/.
