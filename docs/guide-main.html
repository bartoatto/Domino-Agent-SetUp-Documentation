Domino Agentic Coding Setup Guide

A project template guide for configuring AI coding agents with Domino and your domain knowledge

Overview

AI coding agents are agentic assistants that can take actions on your behalf: running workflows, querying systems, analyzing data, and generating outputs. Out of the box, they have no knowledge of Domino, your internal tooling, or your domain.

This guide shows how to close that gap using Domino's infrastructure --- environments, project templates, Git integration, and imported code repositories --- to deliver context and configuration to whichever coding agent your team uses.

Why the Domino plugin matters

Without it, your coding agent has no knowledge of how Domino works. The plugin gives it everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why adding your own skills makes it more powerful

The plugin covers the platform. Your skills cover your domain. When you add skills for your data pipelines, internal APIs, and compliance rules, the agent stops being a generic assistant and starts understanding how your team specifically works.

Two layers, in short

Layer Source What it covers
Domino platform plugin Provided by Domino Jobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing
Your domain skills Added by your team Your workflows, data pipelines, internal APIs, compliance rules

Part 1: The Universal Pattern

Every modern coding agent follows the same fundamental architecture for consuming context:

1. A system prompt file at the repo root

A Markdown file read automatically at session start. It tells the agent what it is, what it can do, and how to behave. The file name varies by agent, but the purpose is identical.

Agent System Prompt File
Claude Code CLAUDE.md
Codex AGENTS.md
Cursor .cursor/rules/*.mdc
Kiro .kiro/steering/*.md
GitHub Copilot .github/copilot-instructions.md

2. MCP (Model Context Protocol) servers

MCP is an open standard for giving agents runtime tools. All major coding agents support it. A single MCP server (like the Domino MCP server) can provide tools to any agent --- the server code is reusable; only the configuration file differs.

Agent MCP Config File Root Key
Claude Code .mcp.json mcpServers
Codex .codex/config.toml [mcp_servers.<id>]
Cursor .cursor/mcp.json mcpServers
Kiro .kiro/settings/mcp.json mcpServers
GitHub Copilot (VS Code) .vscode/mcp.json servers
GitHub Copilot (CLI) ~/.copilot/mcp-config.json mcpServers

3. Skills / rules directories

Modular, topic-specific instruction files the agent loads on demand. The format varies, but the concept is the same: a description tells the agent when to load it, and the body contains the instructions.

Agent Skills Directory File Format
Claude Code .claude/skills/*/ SKILL.md with YAML frontmatter
Codex .agents/skills/*/ SKILL.md with YAML frontmatter
Cursor .cursor/rules/ *.mdc with YAML frontmatter
Kiro .kiro/skills/*/ SKILL.md with YAML frontmatter
GitHub Copilot .github/instructions/ *.instructions.md with applyTo frontmatter

Claude Code, Codex, and Kiro all follow the open Agent Skills standard, making skills portable between them.

4. The delivery mechanism: Domino

The agent-specific files are just configuration. Domino provides the infrastructure to deliver them consistently:

  • Environments install the agent runtime (CLI binary, dependencies)
  • Project Templates deliver the configuration files and directory structure
  • Imported Code Repositories share skills and plugins across every project
  • Pre-run scripts wire everything together at workspace start

Change the template once; every new project gets the updated configuration.

Part 2: Domino Infrastructure Setup

This section is agent-agnostic. The environment installs whichever agent your team uses; the template delivers the corresponding configuration files.

Part 2a: Setting Up the Environment

The workspace environment installs system utilities and MCP server dependencies at build time. The agent runtime is either baked into the image (for air-gapped environments) or installed fresh at workspace start via the pre-run script (for environments with internet access).

Step 1: Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions.

Option A: Open internet --- agent installed at workspace start, always up to date:

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim iptables ipset && \
    rm -rf /var/lib/apt/lists/*

# -- Create directories --
RUN mkdir -p /home/ubuntu/.claude /commandhistory && \
    chown -R ubuntu:ubuntu /home/ubuntu/.claude /commandhistory

# -- Set environment variables --
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"

# -- Install MCP server dependencies (required) --
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional -- add what your team needs) --
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Option B: Air-gapped / restricted internet --- agent baked into the image:

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim iptables ipset && \
    rm -rf /var/lib/apt/lists/*

# -- Create directories --
RUN mkdir -p /home/ubuntu/.claude /commandhistory && \
    chown -R ubuntu:ubuntu /home/ubuntu/.claude /commandhistory

# -- Install the agent into the image --
USER ubuntu
# Claude Code:
RUN curl -fsSL https://claude.ai/install.sh | bash
# Codex (requires Node.js 22+):
# RUN npm install -g @openai/codex
ENV PATH="/home/ubuntu/.claude/bin:/home/ubuntu/.local/bin:$PATH"

# -- Install MCP server dependencies (required) --
USER root
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional) --
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Base image requirements: The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement. Adapt if using a different base user.

Note on IDE-based agents (Cursor, Kiro, Copilot): These agents run as extensions inside their respective IDEs rather than as standalone CLI tools. The Dockerfile above is designed for CLI agents (Claude Code, Codex). For IDE-based agents, the agent itself does not need to be installed in the environment --- it ships with the IDE. The MCP server dependencies and system utilities are still required. See the agent-specific appendices for details.

Step 2: Add the Pre-Run Script

The pre-run script runs automatically at every workspace start. It handles: installing the agent if not present, setting up persistent storage, symlinking plugin skills, and populating project settings.

#!/bin/bash
# start_workspace.sh -- Domino pre-run script for AI coding agents
# Handles installation, per-user persistence, and plugin discovery.
# Run once automatically at workspace start.
#
# Security: OAuth credentials (~/.claude/.credentials.json) are scrubbed
# on each cold start to prevent token leakage via shared /mnt/artifacts.
# Preferences (~/.claude.json) are safe to persist -- they contain no secrets.

set -e

echo "=== Coding Agent Workspace Setup ==="

# ---------------------------------------------------------------
# 1. Install agent if not present
# ---------------------------------------------------------------
# Uncomment the block for your agent:

# -- Claude Code --
if ! command -v claude &> /dev/null; then
    echo "[!] Claude Code not found. Installing..."
    curl -fsSL https://claude.ai/install.sh | bash
    export PATH="$HOME/.local/bin:$PATH"
fi
echo "[ok] Claude Code version: $(claude --version 2>/dev/null || echo 'installed')"

# -- Codex (requires Node.js 22+) --
# if ! command -v codex &> /dev/null; then
#     echo "[!] Codex not found. Installing..."
#     npm install -g @openai/codex
# fi
# echo "[ok] Codex version: $(codex --version 2>/dev/null || echo 'installed')"

# ---------------------------------------------------------------
# 2. Per-user persistent storage (Claude Code specific)
# ---------------------------------------------------------------
if [ -n "$DOMINO_USER_NAME" ]; then
    PERSIST_USER="$DOMINO_USER_NAME"
else
    PERSIST_USER="$USER"
    echo "[!] DOMINO_USER_NAME not set, falling back to: $PERSIST_USER"
fi

PERSIST_DIR="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude"
mkdir -p "$PERSIST_DIR"
echo "[i] Persistent storage: $PERSIST_DIR"

# -- Symlink ~/.claude to persistent storage --
HOME_CLAUDE="$HOME/.claude"

if [ -d "$HOME_CLAUDE" ] && [ ! -L "$HOME_CLAUDE" ]; then
    echo "[i] Migrating existing ~/.claude to persistent storage..."
    cp -r "$HOME_CLAUDE/"* "$PERSIST_DIR/" 2>/dev/null || true
    rm -rf "$HOME_CLAUDE"
fi

[ -L "$HOME_CLAUDE" ] && [ ! -e "$HOME_CLAUDE" ] && rm -f "$HOME_CLAUDE"

if [ ! -L "$HOME_CLAUDE" ]; then
    ln -s "$PERSIST_DIR" "$HOME_CLAUDE"
    echo "[ok] ~/.claude symlinked to $PERSIST_DIR"
else
    echo "[ok] ~/.claude persistence already configured"
fi

# -- Force fresh authentication --
rm -f "$HOME/.claude/.credentials.json" 2>/dev/null || true
rm -f "$HOME/.claude/backups/.claude.json.backup."* 2>/dev/null || true
echo "[ok] Cleared OAuth credentials"

# -- Persist ~/.claude.json (preferences, not secrets) --
PERSIST_JSON="/mnt/artifacts/.claude-users/${PERSIST_USER}/.claude.json"
HOME_JSON="$HOME/.claude.json"

if [ ! -f "$PERSIST_JSON" ]; then
    if [ -f "$HOME_JSON" ] && [ ! -L "$HOME_JSON" ]; then
        cp "$HOME_JSON" "$PERSIST_JSON"
    else
        echo '{}' > "$PERSIST_JSON"
    fi
fi

rm -f "$HOME_JSON" 2>/dev/null || true

if [ ! -L "$HOME_JSON" ]; then
    ln -s "$PERSIST_JSON" "$HOME_JSON"
    echo "[ok] ~/.claude.json symlinked to $PERSIST_JSON"
else
    echo "[ok] ~/.claude.json persistence already configured"
fi

# ---------------------------------------------------------------
# 3. Register Domino plugin (skills, commands, agents)
# ---------------------------------------------------------------
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

# -- Claude Code / Kiro skills (SKILL.md format, Agent Skills standard) --
mkdir -p /mnt/code/.claude/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.claude/skills/$skill_name" 2>/dev/null || true
done

# -- Kiro skills (same standard, different path) --
mkdir -p /mnt/code/.kiro/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done

# -- Codex skills (same standard, different path) --
mkdir -p /mnt/code/.agents/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

# -- Claude Code commands --
mkdir -p /mnt/code/.claude/commands
for cmd_file in "$PLUGIN_DIR"/commands/*.md; do
    [ -f "$cmd_file" ] || continue
    ln -sf "$cmd_file" "/mnt/code/.claude/commands/$(basename "$cmd_file")" 2>/dev/null || true
done

# -- Claude Code agents --
mkdir -p /mnt/code/.claude/agents
for agent_file in "$PLUGIN_DIR"/agents/*.md; do
    [ -f "$agent_file" ] || continue
    ln -sf "$agent_file" "/mnt/code/.claude/agents/$(basename "$agent_file")" 2>/dev/null || true
done

# -- Claude Code output styles --
mkdir -p /mnt/code/.claude/output-styles
for style_file in "$PLUGIN_DIR"/output-styles/*.md; do
    [ -f "$style_file" ] || continue
    ln -sf "$style_file" \
      "/mnt/code/.claude/output-styles/$(basename "$style_file")" 2>/dev/null || true
done

echo "[ok] Registered skills: $(ls /mnt/code/.claude/skills/ 2>/dev/null | wc -l) (Claude Code), \
$(ls /mnt/code/.kiro/skills/ 2>/dev/null | wc -l) (Kiro), \
$(ls /mnt/code/.agents/skills/ 2>/dev/null | wc -l) (Codex)"

# ---------------------------------------------------------------
# 4. Populate domino_project_settings.md from environment
# ---------------------------------------------------------------
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

echo "=== Coding agent setup complete ==="

Part 2b: Setting Up the Template

The template is a Domino project template backed by a Git repository. When a team member creates a new project from the template, they get the configuration files for their chosen agent pre-loaded.

Step 1: Create the Git Repository

Create a new Git repository containing configuration files for all agents your team uses. A multi-agent template looks like this:

/mnt/code/ ├── domino_project_settings.md # Shared: project name/user (read by MCP server) │ ├── CLAUDE.md # Claude Code: system prompt ├── .mcp.json # Claude Code: MCP config ├── .claude/ │ └── settings.json # Claude Code: plugin reference │ ├── AGENTS.md # Codex: system prompt ├── .codex/ │ └── config.toml # Codex: MCP + settings │ ├── .cursor/ │ ├── rules/ │ │ └── domino.mdc # Cursor: system prompt as rule │ └── mcp.json # Cursor: MCP config │ ├── .kiro/ │ ├── steering/ │ │ └── domino.md # Kiro: system prompt as steering │ └── settings/ │ └── mcp.json # Kiro: MCP config │ ├── .github/ │ ├── copilot-instructions.md # Copilot: system prompt │ └── instructions/ │ └── domino.instructions.md # Copilot: Domino-specific instructions ├── .vscode/ │ └── mcp.json # Copilot (VS Code): MCP config

Each agent reads only its own files and ignores the rest. A single template can support multiple agents simultaneously.

The content of each agent's system prompt file is functionally identical --- the same Domino platform rules, data paths, and compliance instructions --- just formatted for each agent's expected file and structure. See the agent-specific appendices for the exact file contents.

Step 2: Register as a Domino Project Template

In Domino, create a new project template pointing to the Git repository. When configuring the template, attach the following repository as Imported Code:

Repository Mounts at
https://github.com/dominodatalab/domino-claude-plugin.git /mnt/imported/code/domino-data-lab-plugin

Also set the default environment to the one built in Part 2a and the default hardware tier.

Step 3: Verify

Create a new project from the template, start an IDE workspace, open a terminal and launch your agent. Try: "What Domino skills do you have?"

The agent should respond with the Domino platform skills from the plugin. If it does, the template is working.

Part 4: Building Your Domain Skills

Out of the box, the template gives the agent knowledge of the Domino platform only. The domain layer --- your workflows, your data, your internal APIs --- is empty. This is where your team's work begins.

The Portable Format: Agent Skills Standard

Claude Code, Codex, and Kiro all follow the open Agent Skills standard. A skill written once in this format works across all three agents:

my-skill/ SKILL.md # Required: instructions + frontmatter scripts/ # Optional: executable code references/ # Optional: documentation assets/ # Optional: templates

SKILL.md format:

---
name: my-skill
description: What this skill covers and when the agent should use it.
---

# My Skill

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes

The description field is critical --- it determines when the agent activates the skill. Write it as a specific, action-oriented sentence.

Cursor and Copilot equivalents

For Cursor, the same content goes into a .mdc file in .cursor/rules/:

---
description: What this rule covers and when Cursor should use it.
globs: "relevant/file/patterns/**"
alwaysApply: false
---

[Same content as SKILL.md body]

For Copilot, the same content goes into a .instructions.md file in .github/instructions/:

---
applyTo: "relevant/file/patterns/**"
---

[Same content as SKILL.md body]

Where to Put Skills

Project-specific skills: Add to the agent's skills directory in your project Git repo. Commit and push; skills persist as long as they're in the repo.

Org-wide skills: Create a dedicated Git repository (e.g., my-org-claude-skills), attach it to the project template as Imported Code, and add a block to the pre-run script to symlink it alongside the Domino plugin:

# In start_workspace.sh, add after the existing plugin symlink loop
ORG_SKILLS="/mnt/imported/code/my-org-claude-skills"

# Agent Skills standard (Claude Code, Codex, Kiro)
for skill_dir in "$ORG_SKILLS"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.claude/skills/$skill_name" 2>/dev/null || true
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
    ln -sf "$skill_dir" "/mnt/code/.agents/skills/$skill_name" 2>/dev/null || true
done

# Cursor rules
for rule_file in "$ORG_SKILLS"/cursor-rules/*.mdc; do
    [ -f "$rule_file" ] || continue
    ln -sf "$rule_file" "/mnt/code/.cursor/rules/$(basename "$rule_file")" 2>/dev/null || true
done

# Copilot instructions
for inst_file in "$ORG_SKILLS"/copilot-instructions/*.instructions.md; do
    [ -f "$inst_file" ] || continue
    ln -sf "$inst_file" "/mnt/code/.github/instructions/$(basename "$inst_file")" 2>/dev/null || true
done

Every project that mounts this repo picks up your org skills automatically on workspace start.

System Prompt vs. Skills

System prompt file (CLAUDE.md / AGENTS.md / steering / instructions): universal rules for every conversation --- platform behavior, path conventions, compliance.

Skills / rules: context-specific knowledge for a particular workflow. If a rule only applies when running a specific pipeline or tool, it belongs in a skill.

Part 5: Extending Further

Custom MCP Servers

Most teams will not need this initially; skills cover a wide range of use cases. Write a custom MCP server when you need the agent to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

A minimal MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("server-1")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def tool_1(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

The MCP server code is fully agent-agnostic --- the same Python server works with every agent. Only the configuration file that points to it differs (see agent appendices for the exact format).

Key principles: one tool per action, precise docstrings (the agent reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Slash Commands (Claude Code specific)

For repetitive multi-step workflows in Claude Code, add slash commands to .claude/commands/. Users type /command-name and Claude expands it to a full set of instructions. Good candidates: initializing new jobs from templates, scaffolding experiments with standard MLflow setup, generating data access request templates.
