GitHub Copilot on Domino

A project template guide for configuring GitHub Copilot with Domino and your domain knowledge

Overview

GitHub Copilot is an AI coding assistant from GitHub. It is available in several forms: a VS Code extension (with agent mode), a JetBrains extension, a CLI (gh copilot), and a cloud agent (the coding agent on github.com). Out of the box, it has no knowledge of Domino, your internal tooling, or your domain.

Already included: GitHub Copilot is pre-installed in the Domino Standard Environment as a VS Code extension. You do not need a custom environment. Users need their own GitHub Copilot subscription (Individual, Business, or Enterprise) to authenticate.

This guide shows how to close that gap. The Domino plugin gives Copilot everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why adding your own instructions makes it more powerful: the plugin covers the platform, but your instructions cover your domain. When you add path-specific instructions for your data pipelines, internal APIs, and compliance rules, Copilot stops being a generic assistant and starts understanding how your team specifically works.

Two layers, in short:

  • The Domino platform plugin: provided by Domino, covers the full platform. This is what you get out of the box.
  • Your instructions: knowledge your team adds on top. Not included in the template. Explained in the Building Your Domain Instructions section.

Official documentation: docs.github.com/en/copilot

Note: The shared setup (Domino plugin, common Dockerfile dependencies, project template concept) is covered on the Overview page. This page covers only the Copilot-specific parts.

Important: Copilot MCP tools are only available in agent mode. You must select "Agent" in the Copilot Chat mode dropdown (VS Code 1.99+). MCP tools are invisible in Ask or Edit mode.

Part 1: How It Fits Together

The template ships with three files plus a shared settings file:

/mnt/code/ ← Your project (Git repo root in Domino) ├── domino_project_settings.md ← Project name/user (read by MCP server) ├── .github/ │ ├── copilot-instructions.md ← System prompt (read every session) │ └── instructions/ │ └── domino.instructions.md ← Path-specific Domino instructions └── .vscode/ └── mcp.json ← Starts the Domino MCP server
FilePurpose
.github/copilot-instructions.md Read automatically at the start of every agent mode session. Tells Copilot what it is, what it can do, and how to behave in this workspace. Plain Markdown, no frontmatter. Update it as your domain knowledge evolves.
.github/instructions/*.instructions.md Path-specific instructions that apply conditionally based on applyTo glob patterns in YAML frontmatter. Use these for domain-specific rules that only matter when working with certain file types or directories.
.vscode/mcp.json Starts the Domino MCP server at workspace launch. Provides runtime tools like run_domino_job, check_domino_job_run_status, file sync utilities, and more. Uses the servers root key (not mcpServers).
domino_project_settings.md Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script using $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME. Users do not need to edit it.

At workspace start, the pre-run script populates domino_project_settings.md from environment variables. The Domino MCP server is started by VS Code when it reads .vscode/mcp.json. Unlike Claude Code, Copilot does not use a plugin system with symlinked skills. Instead, all domain knowledge lives in .github/ as Markdown instruction files.

Part 2: The Shared Domino Plugin

The plugin at /mnt/imported/code/domino-data-lab-plugin is sourced from dominodatalab/domino-claude-plugin. It provides the domino_server MCP server, which gives Copilot runtime tools: run_domino_job, check_domino_job_run_status, check_domino_job_run_results, list_domino_project_files, and file sync utilities.

The MCP server is started by VS Code via .vscode/mcp.json. The plugin also contains skill files, commands, and agents designed for Claude Code. While these are not directly used by Copilot (which has its own instructions system), the MCP server is shared across all agents.

Part 3: Setting Up the Copilot-Domino Template and Environment

Two things need to be created once and reused across all projects: a Domino environment and a Domino project template. The environment provides the runtime: system utilities and MCP server dependencies. The template provides the project structure: the configuration files and imported repositories that Copilot needs to connect to Domino. When a team member creates a new project, they select both; everything comes pre-loaded.

Part 3a: Setting Up the Environment

Copilot is a VS Code extension. It does not need to be installed in the Domino environment. The environment only needs the MCP server dependencies so that the domino_server MCP can run inside the workspace.

Users need (on their local machine):

  • VS Code 1.99+ (required for agent mode with MCP support)
  • GitHub Copilot extension installed in VS Code
  • A GitHub Copilot subscription (Individual, Business, or Enterprise)

Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions. Since Copilot runs in the user's IDE (not in the workspace), the Dockerfile is simpler than for Claude Code. You only need system utilities and MCP dependencies. 

# ── Install system utilities ──
USER root
RUN apt-get update && \
    apt-get install -y jq vim && \
    rm -rf /var/lib/apt/lists/*

# ── Install MCP server dependencies (required) ──
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# ── Install additional packages (optional, add what your team needs) ──
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Base image requirements: the base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement.

Pre-Run Script

The pre-run script runs automatically at every workspace start. For Copilot, it only needs to populate domino_project_settings.md from environment variables. Add this as the workspace pre-run script in your Domino environment or project settings: 

#!/bin/bash
# start_workspace.sh: Domino pre-run script for GitHub Copilot
# Populates project settings from Domino environment variables.
# Run once automatically at workspace start.

set -e

echo "=== Copilot Workspace Setup ==="

# ── 1. Populate domino_project_settings.md from environment ──
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Copilot setup complete ==="

The script is intentionally minimal. Copilot does not need local installation, persistent storage setup, or plugin symlinks. All it needs is the project settings file populated so the MCP server knows which project to operate against.

Part 3b: Setting Up the Template

The template is a Domino project template backed by a git repository. The repository contains the configuration files that wire Copilot to the Domino platform. An additional repository is attached as Imported Code; it mounts at a fixed path that the MCP config references.

Step 1: Create the Git Repository

Create a new git repository containing these files:

/mnt/code/ ├── domino_project_settings.md ├── .github/ │ ├── copilot-instructions.md │ └── instructions/ │ └── domino.instructions.md └── .vscode/ └── mcp.json

Here is what each file should contain (full contents in the Template File Contents section below).

Step 2: Register as a Domino Project Template

In Domino, create a new project template pointing to the git repository you just created. See the Domino documentation on project templates for the full walkthrough.

When configuring the template, attach the following repository as Imported Code:

RepositoryMounts at
https://github.com/dominodatalab/domino-claude-plugin.git /mnt/imported/code/domino-data-lab-plugin

This is the path that .vscode/mcp.json references. Also set the default environment to the one built in Part 3a and the default hardware tier (Small is sufficient for Copilot workspaces).

Step 3: Verify

Create a new project from the template, open it in VS Code with the Copilot extension. Switch to Agent mode (Cmd+L or Ctrl+L, then select "Agent" from the dropdown). Ask: 

What MCP tools do you have available?

Copilot should list the tools from the domino_server MCP server. If it does, the template is working correctly. When team members create new projects from the template, domino_project_settings.md is populated automatically at workspace start. They only need to add their domain-specific instruction files to .github/instructions/.

Gotcha: The first time Copilot uses an MCP server, it asks for trust confirmation. Accept it once and it persists.

Part 4: Template File Contents

.vscode/mcp.json

Starts the Domino MCP server from the imported repo. Environment variables are injected automatically by Domino.

Important: VS Code uses servers as the root key, not mcpServers. This is the most common setup mistake. Copy-pasting a Claude Code or Cursor config without changing the root key means the server will not load.

{
  "servers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

VS Code supports three MCP transport types in .vscode/mcp.json:

TransportFieldUse case
stdiocommand, args, envLocal process (this is what the Domino MCP server uses)
SSEurl, type: "sse"Remote server over Server-Sent Events
HTTP (Streamable)url, type: "http"Remote server over streamable HTTP

Additional optional fields: sandboxEnabled (boolean, controls sandbox mode).

.github/copilot-instructions.md

The persistent system prompt. Copilot instructions are plain Markdown with no frontmatter. This file is read automatically at every agent mode session. It tells Copilot what it is, what it can do, and how to behave. 

# Domino Platform Instructions

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

Trust these instructions. They describe the environment you are operating in.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.vscode/mcp.json`. Available tools:
- `run_domino_job`: run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results`: poll and
  retrieve job output
- `list_domino_project_files`: browse project files
- `sync_local_file_to_domino` / `smart_sync_file` /
  `upload_file_to_domino_project` /
  `download_file_from_domino_project`: file sync
- `get_domino_environment_info`: inspect compute environments
- `open_web_browser`: open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

Content guidelines for copilot-instructions.md

  • High-level repository details: what the project does, languages, frameworks.
  • Build and validation info: bootstrap, build, test, run, lint steps.
  • Project layout and architecture overview.
  • Must not exceed approximately 2 pages of content.
  • Must not be task-specific (use path-specific .instructions.md files for that).
  • Include a trust directive: tell the agent to trust these instructions.

.github/instructions/domino.instructions.md (optional)

Path-specific instructions that apply only when working with certain files. Uses applyTo frontmatter with glob syntax: 

---
applyTo: "**/*.py"
---

When writing Python scripts that will run as Domino jobs:
- Include `#!/usr/bin/env python3` as the first line
- Print results to stdout in plain text tabular format
- Use `/mnt/data/` paths for dataset access, not relative paths
- Log experiment metrics with MLflow when doing model training

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. Users do not need to edit this file. 

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Part 5: Copilot's Custom Instructions System

Copilot does not use the Agent Skills standard (used by Claude Code, Codex, and Kiro). It has its own instructions system with three levels, all of which are combined (not overridden) when active:

LevelSourcePriorityScope
Personal VS Code settings or ~/.copilot/ Highest Individual developer preferences
Repository .github/copilot-instructions.md Medium Project-wide rules (this is what the template provides)
Organization GitHub org settings > Copilot > Custom instructions Lowest Company-wide standards across all repos

All three levels are combined into a single context. Personal instructions take highest priority, but they do not replace the other levels. They are all additive.

Path-Specific Instructions

For domain-specific knowledge that applies only to certain files, create .instructions.md files in .github/instructions/. Each file uses YAML frontmatter with an applyTo field that specifies which files trigger the instructions: 

---
applyTo: "pipelines/**/*.py"
---

# ETL Pipeline Rules

## Overview
Rules for running ETL pipelines on Domino using Spark.

## Key tools
Use the domino_server MCP tools to run pipeline scripts as Domino jobs.

## Rules
- All pipelines read from /mnt/imported/data/warehouse/
- Output to /mnt/data/{PROJECT_NAME}/pipeline-output/
- Log metrics with MLflow

Frontmatter fields

FieldRequiredDescription
applyToYesGlob pattern(s) that trigger these instructions. Multiple patterns: "**/*.ts,**/*.tsx"
nameNoHuman-readable name for the instruction set
descriptionNoBrief description of when these instructions apply
excludeAgentNoExclude from specific agents: "code-review" or "coding-agent"

VS Code settings for instructions

These settings control how Copilot discovers and applies instruction files:

SettingPurpose
chat.includeApplyingInstructionsEnable pattern-based .instructions.md files
chat.instructionsFilesLocationsConfigure where instruction files are searched
github.copilot.chat.organizationInstructions.enabledEnable organization-level instructions

AGENTS.md and CLAUDE.md Support (Experimental)

VS Code Copilot has experimental support for reading AGENTS.md and CLAUDE.md files. These are opt-in and not yet stable:

SettingWhat it enables
chat.useAgentsMdFileRead AGENTS.md for instructions
chat.useNestedAgentsMdFilesSupport nested AGENTS.md files in subdirectories
chat.useClaudeMdFileRead CLAUDE.md for instructions (searches workspace root, .claude/, and ~/)

If your project already has a CLAUDE.md for Claude Code, enabling chat.useClaudeMdFile means Copilot can reuse those same instructions without duplication. This is useful for teams running multiple AI coding agents on the same project.

Part 6: Building Your Domain Instructions

Out of the box, the template gives Copilot knowledge of the Domino platform only. The domain layer (your workflows, your data, your internal APIs) is empty. This is where your team's work begins.

Where to Put Instructions

There are two scopes:

  • Repository-level instructions: the always-on system prompt in .github/copilot-instructions.md. This is read on every agent session. Put universal project rules here.
  • Path-specific instructions: files in .github/instructions/ with applyTo glob patterns. These activate only when Copilot is working with matching files. Put workflow-specific, pipeline-specific, or file-type-specific rules here.

copilot-instructions.md vs. Path-Specific Instructions

FileScopeWhen to use
.github/copilot-instructions.md Universal Rules for every conversation: platform behavior, path conventions, compliance.
.github/instructions/*.instructions.md Context-specific Knowledge for a particular workflow. If a rule only applies when running a specific pipeline or working with specific file types, it belongs in a path-specific instruction file.

Converting Agent Skills to Copilot Instructions

If you have skills in the Agent Skills standard (SKILL.md format, used by Claude Code, Codex, and Kiro), convert to Copilot format by moving the content into a .instructions.md file with appropriate applyTo targeting:

Agent Skills (SKILL.md)Copilot (.instructions.md)
name: etl-pipelineUse filename instead: etl-pipeline.instructions.md
description: Rules for ETL...Replace with applyTo: "pipelines/**/*.py"
Body content (rules, tools, examples)Same body content, no changes needed

Example conversion:

# Agent Skills format: .claude/skills/etl-pipeline/SKILL.md
---
name: etl-pipeline
description: Rules for running ETL pipelines on Domino using Spark.
---
# ETL Pipeline
...rules here...

# Copilot format: .github/instructions/etl-pipeline.instructions.md
---
applyTo: "pipelines/**/*.py"
name: etl-pipeline
description: Rules for running ETL pipelines on Domino using Spark.
---
# ETL Pipeline
...same rules here...

Part 7: The Complete Template

Once the template is set up and the pre-run script has run, a fully built-out project looks like this:

/mnt/code/ ← Your project ├── domino_project_settings.md ← project_name and user_name ├── .github/ │ ├── copilot-instructions.md ← Platform rules, data paths, compliance │ ├── instructions/ │ │ ├── domino.instructions.md ← Domino conventions for Python files │ │ ├── etl-pipeline.instructions.md ← Your workflow │ │ ├── data-sources.instructions.md ← Your data access patterns │ │ ├── internal-apis.instructions.md ← Your proprietary tools and APIs │ │ └── compliance.instructions.md ← Your governance and regulatory rules │ └── agents/ │ └── domino-assistant.md ← Custom agent (optional) └── .vscode/ └── mcp.json ← MCP server config

The .github/copilot-instructions.md file provides the always-on base rules. The path-specific instruction files in .github/instructions/ activate when working with matching file patterns. Your team's domain-specific instructions are committed directly to the project repository.

Part 8: Extending Further

Custom Agents

Copilot supports custom agent profiles in .github/agents/*.md with YAML frontmatter. Custom agents let you create specialized personas with specific tool access and instructions: 

---
name: domino-assistant
description: Helps run Domino jobs and manage experiments
tools:
  - "domino_server/*"
---

You are a Domino platform assistant. Use the domino_server MCP tools
to run jobs, check status, and manage project files.

Always read domino_project_settings.md before making MCP calls.
When running jobs, check status every minute until completion.

Custom agent frontmatter fields

FieldDescription
nameAgent name (how users invoke it)
descriptionWhat the agent does
targetWhere it runs: vscode or github-copilot
toolsList of allowed tools, or "*" for all tools
modelPreferred model to use
mcp-serversWhich MCP servers this agent can access

Custom agents are versioned by git commit SHA, so you get full auditability of agent behavior changes over time.

Custom MCP Servers

Write a custom MCP server when you need Copilot to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog. Register it in .vscode/mcp.json using the servers root key: 

{
  "servers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "my_custom_server": {
      "command": "python",
      "args": [".vscode/mcp-servers/my_server.py"]
    }
  }
}

A minimal MCP server using FastMCP: 

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("my-server")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def query_data_catalog(query: str) -> dict:
    """Query the internal data catalog. Returns matching datasets with locations."""
    resp = requests.post(
        "https://internal-catalog.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Key principles: one tool per action, precise docstrings (Copilot reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Organization-Level Instructions

GitHub organizations can configure Copilot instructions that apply across all repositories. Configure in GitHub.com under Organization settings > Copilot > Custom instructions. These are the lowest priority and are combined with repository and personal instructions.

Use organization-level instructions for company-wide standards: security policies, naming conventions, required compliance checks, approved libraries and frameworks.

Copilot CLI

For terminal use, the gh copilot CLI stores MCP config at ~/.copilot/mcp-config.json. Note: the CLI uses mcpServers as the root key (matching Claude Code), not servers. 

// ~/.copilot/mcp-config.json (CLI format, uses mcpServers)
{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

You can also use the interactive /mcp add command within the CLI to register new MCP servers.

Cloud Coding Agent

The GitHub Copilot coding agent (available on github.com) can also use MCP servers. Configure MCP servers in the repository settings on GitHub. The cloud agent supports secret injection using the ${{ secrets.VAR }} and ${{ vars.VAR }} syntax for environment variables, similar to GitHub Actions.

Diagnostics and Troubleshooting

When things are not working as expected, use these built-in diagnostic tools:

  • Chat Customizations editor: View which instruction files are loaded and their resolved content. Accessible from the Copilot Chat settings.
  • Diagnostics view: Right-click in the Copilot Chat panel to access diagnostics. Shows which MCP servers are connected, which instructions were loaded, and any errors.
  • References section: In any Copilot response, check the References section at the bottom to see which instruction files and context sources were used for that response.

Common issues

SymptomCauseFix
MCP tools not showing up Not in Agent mode Switch to "Agent" in the Copilot Chat dropdown. MCP tools are invisible in Ask or Edit mode.
MCP server not loading Wrong root key in .vscode/mcp.json Use servers, not mcpServers. This is the most common config mistake.
Instructions not applied .instructions.md files not enabled Enable chat.includeApplyingInstructions in VS Code settings.
Trust prompt on every session MCP server trust not persisted Accept the trust prompt once. It should persist after the first confirmation.
Environment variables not injected Workspace not started with correct environment Verify the Domino environment has the pre-run script configured. Check that domino_project_settings.md is populated.
