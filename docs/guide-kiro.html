Domino Kiro Setup Guide

A project template guide for configuring Kiro with Domino and your domain knowledge

Overview

Kiro is Amazon's AI-powered IDE and CLI for agentic software development. It reads steering files from .kiro/steering/ for persistent instructions, loads skills from .kiro/skills/ following the open Agent Skills standard, and connects to external tools via MCP servers configured in .kiro/settings/mcp.json. Kiro also recognizes AGENTS.md at the repository root, making it compatible with projects already configured for Codex or other agents that use the same file.

Why the Domino plugin matters

Without it, Kiro has no knowledge of how Domino works. The plugin gives it everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why steering and skills matter

The plugin covers the platform. Your steering files and skills cover your domain. When you add steering rules for your compliance requirements and skills for your data pipelines, internal APIs, and team conventions, Kiro stops being a generic assistant and starts understanding how your team specifically works.

Two layers of context

Layer Source What it covers
Domino platform plugin Provided by Domino Jobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing
Your domain knowledge Added by your team Your workflows, data pipelines, internal APIs, compliance rules

Three context mechanisms

Kiro supports three complementary ways to receive persistent instructions:

  • Steering files (.kiro/steering/*.md) — always-on rules with conditional inclusion modes. This is Kiro's primary configuration mechanism.
  • Agent Skills (.kiro/skills/*/SKILL.md) — on-demand, topic-specific knowledge loaded when relevant. Follows the same standard as Claude Code and Codex, making skills portable.
  • AGENTS.md (repo root) — a universal system prompt file also read by Codex. Kiro loads it automatically, but it does not support Kiro's inclusion modes.

Part 1: How It Fits Together

The Kiro-Domino integration uses a project template backed by a Git repository, the shared Domino plugin delivered as an imported code repository, and a pre-run script that wires everything together at workspace start. Here is the complete file layout that the template delivers:

/mnt/code/ ├── AGENTS.md ← System prompt (Kiro reads this standard) ├── domino_project_settings.md ← Project name/user (read by MCP server) └── .kiro/ ├── steering/ │ └── domino.md ← Domino platform rules (always loaded) ├── skills/ ← Symlinked from Domino plugin └── settings/ └── mcp.json ← MCP server config

Skills are symlinked from the imported plugin repository into .kiro/skills/ at workspace start by the pre-run script. The plugin follows the open Agent Skills standard (SKILL.md files), which is directly compatible with Kiro's skills path. This is the same mechanism used for Claude Code and Codex — the same skill files, just symlinked to each agent's expected directory.

The plugin repository

The Domino plugin lives in an imported code repository mounted at:

/mnt/imported/code/domino-data-lab-plugin/ ├── skills/ ← Agent Skills (SKILL.md format) │ ├── jobs/SKILL.md │ ├── environments/SKILL.md │ ├── datasets/SKILL.md │ ├── workspaces/SKILL.md │ ├── app-deployment/SKILL.md │ ├── model-endpoints/SKILL.md │ ├── experiment-tracking/SKILL.md │ ├── genai-tracing/SKILL.md │ ├── distributed-computing/SKILL.md │ └── ... ├── mcp-servers/ │ └── domino_mcp_server/ │ └── domino_mcp_server.py ← Agent-agnostic MCP server └── README.md

How Kiro loads context

  1. Steering files in .kiro/steering/ load according to their inclusion mode (always, fileMatch, manual, or auto)
  2. AGENTS.md at the repo root is read automatically at session start
  3. MCP servers configured in .kiro/settings/mcp.json start and register their tools
  4. Skills from .kiro/skills/ are discovered at startup (names and descriptions only); full content loads on demand when the request matches the description

Priority

  • Workspace steering (.kiro/steering/) overrides global steering (~/.kiro/steering/)
  • Workspace skills (.kiro/skills/) override global skills (~/.kiro/skills/)

Part 2: The Shared Domino Plugin

The Domino plugin is the same shared component used across all supported coding agents. It provides two things: a set of skills that teach the agent how to use Domino features, and an MCP server that gives the agent runtime tools to take actions on the platform.

Skills follow the Agent Skills standard

Each skill is a directory containing a SKILL.md file with YAML frontmatter (name, description) and markdown body (instructions, examples, rules). This is the same open standard used by Claude Code and Codex. Kiro reads skills from .kiro/skills/*/SKILL.md — and since the format is identical, the pre-run script simply symlinks the plugin's skill directories into that path.

No conversion or adaptation is needed. A skill written once works with Claude Code, Codex, and Kiro without modification.

MCP server is agent-agnostic

The Domino MCP server (domino_mcp_server.py) is a standard MCP server built with FastMCP. It exposes the same tools regardless of which agent calls them. The server code is identical for every agent; only the configuration file that points to it differs.

Available MCP tools

Tool Purpose
run_domino_job Run a script as a governed Domino job
check_domino_job_run_status Poll job execution status
check_domino_job_run_results Retrieve job output and logs
list_domino_project_files Browse project files on the platform
sync_local_file_to_domino Push a local file to Domino
smart_sync_file Intelligent file sync (detects direction)
upload_file_to_domino_project Upload file to project storage
download_file_from_domino_project Download file from project storage
get_domino_environment_info Inspect compute environments
open_web_browser Open Domino URLs in the browser

Part 3: Setting Up the Kiro-Domino Template and Environment

Part 3a: Environment

Kiro is primarily an IDE that runs on the user's machine (download from kiro.dev). It also has a CLI for headless or terminal-based use. Unlike CLI-only agents such as Claude Code or Codex, the Kiro IDE itself does not need to be installed in the Domino environment — it connects remotely to the workspace.

However, the Domino environment still needs:

  • MCP server dependencies — the Domino MCP server runs inside the workspace regardless of how the user connects
  • System utilities — tools like ripgrep, jq, and vim that the agent may invoke

Dockerfile instructions

# -- Install system utilities --
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*

# -- Install MCP server dependencies (required) --
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# -- Install additional packages (optional -- add what your team needs) --
RUN pip install pandas pyarrow scikit-learn mlflow
USER ubuntu

Note: Since Kiro is an IDE-based agent, the agent runtime does not need to be installed in the Dockerfile. The MCP server and its dependencies are the only required components in the environment.

Pre-run script

The pre-run script runs automatically at every workspace start. For Kiro, it handles: symlinking plugin skills to .kiro/skills/ and populating domino_project_settings.md with real values from environment variables. 

#!/bin/bash
# start_workspace.sh -- Domino pre-run script for Kiro
set -e

echo "=== Kiro Workspace Setup ==="

# ---------------------------------------------------------------
# 1. Symlink Domino plugin skills into .kiro/skills/
# ---------------------------------------------------------------
PLUGIN_DIR="/mnt/imported/code/domino-data-lab-plugin"

mkdir -p /mnt/code/.kiro/skills
for skill_dir in "$PLUGIN_DIR"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done

echo "[ok] Registered skills: $(ls /mnt/code/.kiro/skills/ 2>/dev/null | wc -l) (Kiro)"

# ---------------------------------------------------------------
# 2. Populate domino_project_settings.md from environment
# ---------------------------------------------------------------
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

echo "=== Kiro workspace setup complete ==="

Part 3b: Template

Create a Git repository containing the Kiro configuration files. When a team member creates a new Domino project from this template, they get the full Kiro-Domino integration pre-loaded.

File: .kiro/settings/mcp.json

Configures the Domino MCP server. Environment variables are injected automatically by Domino at workspace start: 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

File: .kiro/steering/domino.md

The core Domino platform rules as a Kiro steering file. The YAML frontmatter sets inclusion: always, ensuring these rules load in every interaction: 

---
inclusion: always
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.kiro/settings/mcp.json`. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve job output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` /
  `download_file_from_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

Steering inclusion modes

Kiro steering files support four inclusion modes via YAML frontmatter. Choose the mode that matches how the content should be loaded:

Mode Frontmatter When loaded
Always inclusion: always Every interaction — use for core platform rules
File Match inclusion: fileMatch + fileMatchPattern: "*.py" When active files match the glob pattern
Manual inclusion: manual Only when the user types #steering-file-name
Auto inclusion: auto + name + description Agent decides based on description match to the current request

The Domino platform steering file uses always because the platform rules must be present in every conversation. Domain-specific steering files can use other modes to keep context lean.

Optional: Foundation steering files

Kiro recommends three additional foundation steering files for project context. These are optional but help Kiro make better suggestions aligned with your stack:

.kiro/steering/product.md — defines your product's purpose:

---
inclusion: always
---

# Product Overview

[Your product description, target users, key features, business objectives]

.kiro/steering/tech.md — documents your technology stack:

---
inclusion: always
---

# Technology Stack

- **Platform**: Domino Data Lab
- **Languages**: Python 3.10+
- **ML Frameworks**: scikit-learn, PyTorch, TensorFlow
- **Experiment Tracking**: MLflow (Domino Experiment Manager)

.kiro/steering/structure.md — describes your project layout:

---
inclusion: always
---

[Your directory layout, key files, naming conventions]

File: AGENTS.md

Kiro also reads AGENTS.md at the repo root. If you already have an AGENTS.md for Codex or other agents, Kiro will read it automatically — no duplication needed. However, AGENTS.md does not support Kiro's inclusion modes; it always loads fully.

If your template supports multiple agents, keep the core Domino rules in both .kiro/steering/domino.md and AGENTS.md. Each agent reads only its own files and ignores the rest.

File: domino_project_settings.md

Same as all other agents. Auto-populated at workspace start by the pre-run script: 

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Imported code repository

Attach the Domino plugin repository as Imported Code in the project template:

Repository Mounts at
https://github.com/dominodatalab/domino-claude-plugin.git /mnt/imported/code/domino-data-lab-plugin

Register as a Domino project template

In Domino, create a new project template pointing to the Git repository. Set the default environment to the one built in Part 3a and attach the imported code repository above.

Verify

Create a new project from the template, start a workspace, and open Kiro (IDE or CLI). Ask: 

What Domino skills do you have?

Kiro should list the Domino platform skills discovered from .kiro/skills/. If it does, the setup is working.

Part 4: Building Your Domain Skills

Out of the box, the template gives Kiro knowledge of the Domino platform only. The domain layer — your workflows, your data, your internal APIs — is empty. This is where your team's work begins.

Two formats for domain knowledge

Kiro supports two complementary formats for adding domain knowledge:

Format Location When loaded Best for
Steering files .kiro/steering/*.md Based on inclusion mode (always, fileMatch, manual, auto) Universal rules, compliance requirements, conventions that apply broadly
Agent Skills .kiro/skills/*/SKILL.md On demand, when the request matches the skill description Topic-specific knowledge for particular workflows, tools, or pipelines

Steering vs. Skills: when to use which

  • Steering: rules that should always be present or conditionally present based on file context. Examples: "always commit before running jobs," "use MLflow for experiment tracking," "Python files must include type hints."
  • Skills: detailed knowledge for a specific workflow that only matters when the user is working on that topic. Examples: an ETL pipeline skill, a model deployment skill, a compliance review skill.

The portable format: Agent Skills standard

Claude Code, Codex, and Kiro all follow the open Agent Skills standard. A skill written once in this format works across all three agents without modification:

my-skill/ SKILL.md # Required: instructions + frontmatter scripts/ # Optional: executable code references/ # Optional: documentation assets/ # Optional: templates

SKILL.md format:

---
name: my-skill
description: What this skill covers and when the agent should use it.
---

# My Skill

## Overview
Brief description of what this skill is for.

## Key tools
List the MCP tools or packages the agent should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes

The description field is critical — it determines when the agent activates the skill. Write it as a specific, action-oriented sentence.

Steering file examples

A domain steering file using fileMatch to load only when Python files are active:

---
inclusion: fileMatch
fileMatchPattern: "pipelines/**/*.py"
---

# ETL Pipeline Rules

- All pipeline scripts must log to /mnt/data/pipeline-logs/
- Use the domino-data SDK for data source connections
- Always validate schema before writing output
- Include row counts in stdout for result verification

A domain steering file using auto mode, where the agent decides based on context:

---
inclusion: auto
name: compliance-rules
description: Data governance and compliance rules for regulated datasets
---

# Compliance Rules

- PII columns must be masked before any export
- All model outputs require an audit trail via MLflow
- Dataset access must be logged with user and timestamp

Where to put skills

Project-specific skills: Add directly to .kiro/skills/ in your project Git repo. Commit and push; skills persist as long as they are in the repo.

Org-wide skills: Create a dedicated Git repository (e.g., my-org-kiro-skills), attach it to the project template as Imported Code, and add a block to the pre-run script to symlink it alongside the Domino plugin: 

# In start_workspace.sh, add after the existing plugin symlink loop
ORG_SKILLS="/mnt/imported/code/my-org-kiro-skills"

for skill_dir in "$ORG_SKILLS"/skills/*/; do
    skill_name=$(basename "$skill_dir")
    ln -sf "$skill_dir" "/mnt/code/.kiro/skills/$skill_name" 2>/dev/null || true
done

Every project that mounts this repo picks up your org skills automatically on workspace start.

Sharing skills across agents

Since Claude Code, Codex, and Kiro all follow the Agent Skills standard, the same SKILL.md files work for all three. The pre-run script symlinks them into each agent's expected path: 

# Same SKILL.md files, different paths
ln -sf "$PLUGIN_DIR/skills/$skill_name" "/mnt/code/.claude/skills/$skill_name"
ln -sf "$PLUGIN_DIR/skills/$skill_name" "/mnt/code/.kiro/skills/$skill_name"
ln -sf "$PLUGIN_DIR/skills/$skill_name" "/mnt/code/.agents/skills/$skill_name"

Part 6: Extending Further

Custom MCP servers

The Domino MCP server covers platform operations. When you need the agent to take actions against an internal system at runtime — querying a proprietary database, submitting to an internal API, fetching from a data catalog — write a custom MCP server. The server code is fully agent-agnostic; register it in .kiro/settings/mcp.json alongside the Domino server: 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "my_internal_server": {
      "command": "python",
      "args": ["/mnt/imported/code/my-org-tools/mcp-servers/internal_server.py"],
      "env": {
        "INTERNAL_API_KEY": "${MY_INTERNAL_API_KEY}"
      }
    }
  }
}

A minimal custom MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("my-internal-server")

@mcp.tool()
def query_data_catalog(search_term: str) -> dict:
    """Search the internal data catalog. Returns matching datasets with locations."""
    resp = requests.post(
        "https://data-catalog.myorg.net/api/search",
        json={"query": search_term},
        headers={"Authorization": f"Bearer {os.environ['INTERNAL_API_KEY']}"},
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Agent hooks

Kiro supports event-driven hooks that execute automatically in response to events in the development workflow. Hooks can be triggered by:

  • File events — when files are created, modified, or deleted
  • Prompt lifecycle — before or after the agent processes a request

This is a Kiro-specific capability not found in Claude Code or Codex. Hooks are useful for enforcing standards automatically (e.g., running a linter when a Python file is saved, updating documentation when an API file changes).

Invoking skills explicitly

In Kiro's chat interface, you can invoke a skill directly by typing /skill-name. This is useful when you want to force a specific skill to load rather than waiting for the agent to decide. Kiro does not use slash commands in the same way as Claude Code — the / prefix invokes skills explicitly.
