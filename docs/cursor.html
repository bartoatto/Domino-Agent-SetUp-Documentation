Cursor on Domino

Agent-specific configuration for Cursor

Cursor is an AI code editor built as a fork of VS Code. It is an IDE-based agent, it runs on the user's machine and connects to a Domino workspace remotely. It does not need to be installed in the Domino environment.

This page covers the Cursor-specific template files. For shared infrastructure (the Domino plugin, environment setup, MCP server dependencies, building domain skills, and custom MCP servers), see the Overview.

Template Files

A Cursor-ready Domino project requires these files in the project git repository:

/mnt/code/ ├── domino_project_settings.md ← Project name/user (shared) └── .cursor/ ├── rules/ ┌ └── domino.mdc ← Domino platform rules (always applied) └── mcp.json ← MCP server config

Environment Setup

Cursor is a desktop IDE, no install is needed in the Domino environment. Users install Cursor on their own machine and connect to the Domino workspace. The only environment requirement is the MCP server dependencies (fastmcp, requests, python-dotenv), which are covered in the common Dockerfile base on the Overview page.

Pre-Run Script

Since Cursor does not need to be installed in the environment and has no plugin symlinks to wire up, the pre-run script only needs to populate domino_project_settings.md:

#!/bin/bash
set -e
echo "=== Cursor Workspace Setup ==="

# Populate domino_project_settings.md
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Cursor setup complete ==="

Template File Contents

.cursor/mcp.json

Starts the Domino MCP server from the imported repo. Environment variables are injected automatically by Domino:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

.cursor/rules/domino.mdc

Domino platform rules as a Cursor rule file. This is the equivalent of CLAUDE.md for Claude Code. The alwaysApply: true frontmatter ensures it is loaded in every conversation:

---
description: Domino Data Lab platform rules for running jobs, accessing data, and deploying apps.
alwaysApply: true
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get the
`project_name` and `user_name`. If they contain placeholder values, immediately
update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.cursor/mcp.json`. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather than
  in the local terminal. This ensures governance, reproducibility, and audit trails.
- Read domino_project_settings.md before making any MCP tool calls.
- When running a job, always check its status and results if completed, and
  briefly explain any conclusions from the result.
- If a job result includes an MLflow or experiment run URL, share that URL.

## Data Access

- Project data is accessible under /mnt/data/ or /mnt/imported/data/.
- Before using a dataset, run a job to list folder contents recursively.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to stdout.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- Commit and push changes before running any Domino jobs, because Domino
  executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino's proxy.
- Create an app.sh as the launch file entry point.

Cursor Rule Types

Cursor supports four types of rules, controlled by YAML frontmatter in .mdc files. Each type determines when and how the rule is loaded:

Type Frontmatter When loaded
Always alwaysApply: true Every conversation
Auto Attached globs: "*.py" When referenced files match the glob pattern
Agent Requested description: "..." (no globs, no alwaysApply) Agent decides based on description
Manual (none of the above) Only when user types @ruleName

Recommendation: Use Always for Domino platform rules (they apply to every interaction). Use Agent Requested or Auto Attached for domain skills that are only relevant in specific contexts.

Skills Format

Cursor does not use the open Agent Skills standard (SKILL.md). Instead, each skill is a .mdc file in .cursor/rules/ with YAML frontmatter.

Converting from Agent Skills to .mdc

If you have existing skills in the Agent Skills format (used by Claude Code, Codex, and Kiro), convert them as follows:

Agent Skills format (SKILL.md)

---
name: etl-pipeline
description: ETL pipeline workflows for data ingestion and transformation.
---

# ETL Pipeline
...

Cursor format (etl-pipeline.mdc)

---
description: ETL pipeline workflows for data ingestion and transformation.
alwaysApply: false
---

# ETL Pipeline
...

Key differences:

  • Drop the name field, the filename serves as the identifier.
  • Add alwaysApply, globs, or neither to control when the rule is loaded (see the rule types table above).
  • The body content (Markdown) stays the same.

Complete Template

A fully built-out Cursor project with both platform rules and domain skills looks like this:

/mnt/code/ ├── domino_project_settings.md └── .cursor/ ├── mcp.json └── rules/ ├── domino.mdc ← Platform rules (always) ├── etl-pipeline.mdc ← your workflow ├── data-sources.mdc ← your data patterns └── compliance.mdc ← your governance rules

Extending

Custom MCP servers

Add additional servers to .cursor/mcp.json under the mcpServers key, the format is the same as Claude Code's .mcp.json:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "your_custom_server": {
      "command": "python",
      "args": ["path/to/your_server.py"]
    }
  }
}

No slash commands

Cursor does not have a slash-command equivalent like Claude Code's .claude/commands/. For reusable multi-step workflows, create Agent Requested rules instead, rules with only a description in the frontmatter that the agent loads when relevant.

Verification

To verify the setup is working:

  1. Open the project in Cursor.
  2. Switch to Agent mode (Cmd+L on macOS, Ctrl+L on Windows/Linux).
  3. Ask: "What MCP tools do you have available?"

Cursor should list the domino_server tools (run_domino_job, check_domino_job_run_status, etc.). If it does, the MCP connection is working. Try a follow-up: "List the files in my Domino project" to confirm end-to-end connectivity.
