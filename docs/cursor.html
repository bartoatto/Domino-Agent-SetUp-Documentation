Cursor on Domino

Agent-specific configuration for Cursor

Cursor is an AI code editor built as a fork of VS Code. It is a standalone desktop IDE (it does not require a separate VS Code installation) that runs on the user's local machine and connects to a Domino workspace remotely. Cursor can take actions on your behalf: running workflows, querying systems, analyzing data, and generating outputs. Out of the box, it has no knowledge of Domino, your internal tooling, or your domain. This page shows how to close that gap.

Note: The shared setup, the Domino plugin, common Dockerfile dependencies, and the project template concept, is covered on the Overview page. This page focuses on the Cursor-specific configuration.

Part 1: How It Fits Together

Cursor connects to Domino through two mechanisms: rules that tell the AI how to behave on the platform, and an MCP server that gives it runtime tools to take actions (run jobs, sync files, check results). Both are stored in the project's .cursor/ directory and committed to Git so every team member gets the same setup.

The template ships with three files:

/mnt/code/ ├── domino_project_settings.md ← Project name/user (shared across agents) └── .cursor/ ├── mcp.json ← Starts the Domino MCP server └── rules/ └── domino.mdc ← Domino platform rules (always applied)
FilePurpose
.cursor/rules/domino.mdc The equivalent of Claude Code's CLAUDE.md. Read at the start of every Cursor session when alwaysApply: true is set. Tells the AI what it is, what it can do, and how to behave on Domino.
.cursor/mcp.json Starts the Domino MCP server when Cursor connects. Provides runtime tools like run_domino_job, check_domino_job_run_status, file sync utilities, and more.
domino_project_settings.md Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script. Shared with all other agents.

Unlike Claude Code, Cursor does not use a plugin system with symlinked skills, commands, or agents. Instead, all Domino platform knowledge goes directly into rule files under .cursor/rules/. Domain knowledge is added as additional rule files in the same directory.

Part 2: About Cursor

What It Is

Cursor is a standalone AI code editor forked from Visual Studio Code. It runs natively on Windows, macOS, and Linux. Download it from cursor.com. System requirements: 4 GB or more of RAM and a stable internet connection. It does not need to be installed in the Domino environment; it runs entirely on the user's machine and connects to Domino workspaces via SSH or the Domino proxy.

System Prompt

Cursor reads its system prompt from rules stored in the .cursor/rules/ directory at the project root. There are two supported file formats:

  • Current format (v2.2 and later): Folder-based rules. Each rule is a directory under .cursor/rules/ containing a RULE.md file, with optional scripts/ and templates/ subdirectories.
  • Legacy format: Single .mdc files in .cursor/rules/. Still fully functional and widely used.
  • Deprecated: A .cursorrules file at the project root. Still recognized but superseded by the rules directory.

Cursor also supports AGENTS.md at the project root (with nested subdirectory support), providing partial compatibility with the emerging agent configuration standard.

Rule Frontmatter

Each rule file begins with YAML frontmatter that controls when and how the rule is loaded. The available fields are:

FieldTypeDescription
description string Explains what the rule covers. Used by the AI to decide whether to load "Agent Requested" rules.
globs file patterns Glob patterns (e.g., "*.py", ["*.ts", "*.tsx"]) that trigger auto-attachment when matching files are referenced.
alwaysApply boolean When true, the rule is injected into every conversation regardless of context.

Rule Types

The combination of frontmatter fields determines the rule type. There are four types, and each controls when the rule is loaded into the conversation:

TypeFrontmatterWhen LoadedUse Case
Always Apply alwaysApply: true Every conversation, unconditionally Domino platform rules, coding standards, compliance requirements
Auto Attached globs: ["*.py"] When referenced files match the glob pattern Language-specific conventions, framework rules (e.g., apply React rules only when editing .tsx files)
Agent Requested description: "..." only (no globs, no alwaysApply) AI evaluates the description and decides if the rule is relevant to the current task Domain-specific workflows, pipeline instructions, API integration guides
Manual All fields empty or absent Only when the user explicitly types @rule-name Rarely used templates, one-off procedures

Important: Injection does not guarantee activation. A rule can be injected into the conversation context, but the AI may still choose not to follow it if it deems the rule irrelevant to the current task. Rules are suggestions, not hard enforcement. Effectiveness can also vary by model.

Rule Precedence

Cursor supports three scopes of rules. When rules conflict, higher-precedence scopes win:

  1. Team Rules (highest): Configured via the Cursor dashboard. Available on Team and Enterprise plans. Can be enforced organization-wide.
  2. Project Rules: Files in .cursor/rules/, version-controlled with the project. This is where Domino platform rules live.
  3. User Rules (lowest): Global preferences set in Cursor Settings. Apply across all projects for that user.

MCP Support

Cursor supports the Model Context Protocol (MCP) via .cursor/mcp.json. The format uses a mcpServers root key, identical to Claude Code's .mcp.json. MCP servers provide runtime tools that the AI can call during a conversation.

Gotchas

  • The built-in UI for editing rules can be buggy. Use an external text editor for reliability.
  • Rules are suggestions, not enforcement. The AI may ignore a rule if it determines the rule is not relevant.
  • Rule effectiveness varies by model. Test with the model your team actually uses.
  • Cursor does not have a native plugin or slash-command system like Claude Code. Reusable workflows are implemented as Agent Requested rules instead.

Part 3: Environment and Pre-Run Setup

Environment Setup

Cursor is a desktop IDE. No install is needed in the Domino environment. Users install Cursor on their own machine and connect to the Domino workspace. The only environment requirement is the MCP server dependencies (fastmcp, requests, python-dotenv), which are covered in the common Dockerfile base on the Overview page: 

# MCP server dependencies (required)
RUN pip install --no-cache-dir fastmcp requests python-dotenv

No Claude Code installation, no directory creation, no persistent storage setup. The Domino environment only needs to serve as the execution backend for MCP tool calls.

Pre-Run Script

Since Cursor does not need to be installed in the environment and has no plugin symlinks to wire up, the pre-run script only needs to populate domino_project_settings.md from environment variables. This is significantly simpler than the Claude Code pre-run script: 

#!/bin/bash
# start_workspace.sh: Domino pre-run script for Cursor
# Populates project settings from Domino environment variables.
# Run once automatically at workspace start.

set -e

echo "=== Cursor Workspace Setup ==="

# Populate domino_project_settings.md from environment
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated"
fi

echo "=== Cursor setup complete ==="

What the script handles

  1. Populate project settings: writes $DOMINO_PROJECT_NAME and $DOMINO_USER_NAME into domino_project_settings.md so the MCP server knows which project to operate against.

Compare this with the Claude Code pre-run script, which handles seven steps (installation, persistent storage, authentication, preference persistence, plugin registration, and settings population). Cursor's simplicity here is a direct benefit of being a desktop application with no server-side agent to install or configure.

Part 4: Template File Contents

This section shows the exact contents of each template file. Create these in a Git repository and register that repository as a Domino project template (see the Overview page for the full template registration walkthrough).

.cursor/mcp.json

Starts the Domino MCP server from the imported plugin repo. The format uses a mcpServers root key, identical to Claude Code's .mcp.json. Environment variables are injected automatically by Domino: 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

.cursor/rules/domino.mdc

Domino platform rules as a Cursor rule file. This is the equivalent of CLAUDE.md for Claude Code. The alwaysApply: true frontmatter ensures it is injected into every conversation: 

---
description: Domino Data Lab platform rules for running jobs, accessing data, and deploying apps.
alwaysApply: true
---

# Domino Platform Rules

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get the
`project_name` and `user_name`. If they contain placeholder values, immediately
update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.cursor/mcp.json`. Available tools:
- `run_domino_job` -- run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` -- poll and retrieve output
- `list_domino_project_files` -- browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` -- file sync
- `get_domino_environment_info` -- inspect compute environments
- `open_web_browser` -- open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather than
  in the local terminal. This ensures governance, reproducibility, and audit trails.
- Read domino_project_settings.md before making any MCP tool calls.
- When running a job, always check its status and results if completed, and
  briefly explain any conclusions from the result.
- If a job result includes an MLflow or experiment run URL, share that URL.

## Data Access

- Project data is accessible under /mnt/data/ or /mnt/imported/data/.
- Before using a dataset, run a job to list folder contents recursively.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to stdout.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- Commit and push changes before running any Domino jobs, because Domino
  executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino's proxy.
- Create an app.sh as the launch file entry point.

domino_project_settings.md

Auto-populated at workspace start by the pre-run script. Users do not need to edit this file.

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Part 5: Cursor Rule Types in Depth

Rules are Cursor's primary mechanism for customizing AI behavior. Understanding the four types and when to use each is essential for building an effective Domino integration.

Always Apply Rules

Set alwaysApply: true in the frontmatter. These rules are injected into every conversation, every time. Use for:

  • Domino platform rules (the domino.mdc file above)
  • Organization-wide coding standards
  • Compliance and security constraints that must never be skipped
---
description: Always-on Domino platform integration.
alwaysApply: true
---

# Your always-applied content here

Auto Attached Rules (Glob-Based)

Set globs to a file pattern or array of patterns. The rule is automatically injected when the user references a file matching the pattern.

---
description: Python-specific Domino conventions for data scripts.
globs: "*.py"
---

# Python Conventions
- Use pathlib for file paths under /mnt/data/
- Always include error handling for MCP tool calls
- Log outputs to stdout in tabular format

You can also use an array of patterns:

globs: ["*.ts", "*.tsx", "*.jsx"]

Agent Requested Rules

Set only description (no globs, no alwaysApply). The AI reads the description and decides whether the rule is relevant to the current task. This is the closest equivalent to Claude Code's skills.

---
description: ETL pipeline workflows for data ingestion and transformation using Domino Jobs.
---

# ETL Pipeline

## Overview
Covers the standard data ingestion pipeline...

## How to run
1. Create a Python script in /mnt/code/scripts/
2. Use run_domino_job to execute it
3. Check results with check_domino_job_run_results

Manual Rules

Leave all frontmatter fields empty or absent. The rule is only loaded when the user explicitly types @rule-name in the chat.

---
---

# Deployment Checklist
Run through this checklist before deploying to production:
1. All tests pass
2. Model metrics meet threshold
3. Governance bundle is approved

Choosing the Right Type

ScenarioRecommended TypeReasoning
Domino platform behavior Always Apply Must be present in every conversation to ensure jobs run correctly, data paths are correct, and Git discipline is followed.
Python data scripts Auto Attached (*.py) Only relevant when editing Python files. No need to load when working with configuration or documentation.
ETL pipeline instructions Agent Requested Only relevant when the user is working on data ingestion tasks. The AI evaluates the description and loads it when appropriate.
One-off deployment checklist Manual Rarely needed. User explicitly invokes it with @deployment-checklist.

Part 6: Building Domain Skills

Out of the box, the template gives Cursor knowledge of the Domino platform only. The domain layer (your workflows, your data, your internal APIs) is empty. This is where your team's work begins.

Where to Put Skills

All Cursor rules live in .cursor/rules/ within the project repository. Unlike Claude Code, there is no separate plugin directory or symlink mechanism. You have two options for organizing domain rules:

  • Project-specific rules: Add .mdc files (or RULE.md folders) directly to .cursor/rules/ in your project repo. Commit and push to persist them.
  • Org-wide rules: Maintain a shared Git repository of rule files and copy them into project templates. Alternatively, use Cursor's Team Rules feature (available on Team and Enterprise plans) to enforce rules across all team members.

Anatomy of a Cursor Rule

Cursor does not use the open Agent Skills standard (SKILL.md). Instead, each skill is either a .mdc file in .cursor/rules/ or a folder with a RULE.md file. The Markdown body is the same; only the frontmatter differs. 

---
description: What this rule covers and when the AI should use it.
alwaysApply: false
---

# Rule Name

## Overview
Brief description of what this rule is for.

## Key tools
List the MCP tools or packages the AI should use for this domain.

## How to run [your workflow]
A short code example or step-by-step for the most common task.

## Rules
- Data input/output paths
- Naming conventions
- Compliance constraints specific to this workflow

## Common errors and fixes
Known failure modes and how to resolve them.

The description field is what the AI uses to decide when to load the rule (for Agent Requested types). Write it as a specific, action-oriented sentence. Rules load only when the AI determines they are relevant.

Converting from Agent Skills to Cursor Rules

If you have existing skills in the Agent Skills format (used by Claude Code, Codex, and Kiro), convert them to Cursor rules as follows:

Agent Skills format (.claude/skills/etl-pipeline/SKILL.md)

---
name: etl-pipeline
description: ETL pipeline workflows for data ingestion and transformation.
---

# ETL Pipeline

## Overview
Covers the standard data ingestion pipeline...

## Key tools
- run_domino_job
- check_domino_job_run_results

## How to run
1. Create script in /mnt/code/scripts/
2. Run as Domino job
3. Check results

Cursor format (.cursor/rules/etl-pipeline.mdc)

---
description: ETL pipeline workflows for data ingestion and transformation.
alwaysApply: false
---

# ETL Pipeline

## Overview
Covers the standard data ingestion pipeline...

## Key tools
- run_domino_job
- check_domino_job_run_results

## How to run
1. Create script in /mnt/code/scripts/
2. Run as Domino job
3. Check results

Key differences in the conversion

ChangeDetails
Drop the name field The filename serves as the identifier in Cursor. etl-pipeline.mdc becomes the rule name.
Add a loading strategy Set alwaysApply, globs, or leave only description to control when the rule is loaded (see rule types above).
Move the file From .claude/skills/<name>/SKILL.md to .cursor/rules/<name>.mdc.
Body stays the same The Markdown content (everything after the frontmatter) transfers directly without changes.

Using the Folder Format (v2.2 and Later)

For rules that need supporting files (templates, scripts, reference data), use the folder-based format introduced in Cursor v2.2:

.cursor/rules/ └── etl-pipeline/ ├── RULE.md ← The rule content (same frontmatter as .mdc) ├── scripts/ │ └── validate_schema.py ← Helper scripts referenced by the rule └── templates/ └── pipeline_config.yaml ← Template files the AI can use

The RULE.md file uses the same YAML frontmatter as .mdc files. The folder name becomes the rule identifier.

domino.mdc vs. Domain Rules

FileScopeWhen to Use
domino.mdc Universal (Always Apply) Rules for every conversation: platform behavior, path conventions, Git discipline, compliance.
Domain .mdc files Context-specific (Agent Requested or Auto Attached) Knowledge for a particular workflow. If a rule only applies when running a specific pipeline or tool, it belongs in a separate rule file.

Part 7: The Complete Template

A fully built-out Cursor project with both platform rules and domain skills looks like this:

/mnt/code/ ├── domino_project_settings.md ← Project name and user name └── .cursor/ ├── mcp.json ← MCP server config (Domino + custom) └── rules/ ├── domino.mdc ← Platform rules (Always Apply) ├── etl-pipeline.mdc ← Your data ingestion workflow (Agent Requested) ├── data-sources.mdc ← Your data access patterns (Agent Requested) ├── python-standards.mdc ← Python conventions (Auto Attached, globs: "*.py") └── compliance.mdc ← Your governance rules (Always Apply)

The Domino platform rule (domino.mdc) is always loaded. Domain rules are loaded based on their type: Auto Attached when matching files are referenced, Agent Requested when the AI determines they are relevant, or Manual when explicitly invoked.

Part 8: Extending

Custom MCP Servers

Write a custom MCP server when you need Cursor to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog. Register it by adding an entry to the mcpServers key in .cursor/mcp.json: 

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "your_custom_server": {
      "command": "python",
      "args": ["path/to/your_server.py"]
    }
  }
}

The format is identical to Claude Code's .mcp.json. A minimal MCP server using FastMCP: 

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("my-server")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def query_catalog(query: str) -> dict:
    """Query internal data catalog. Returns matching datasets."""
    resp = requests.post(
        "https://catalog.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Key principles: one tool per action, precise docstrings (Cursor reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Agent Requested Rules as Workflow Shortcuts

Cursor does not have a slash-command system like Claude Code's .claude/commands/. For reusable multi-step workflows, create Agent Requested rules instead. Write a clear description so the AI loads the rule when the user asks about that topic: 

---
description: Standard procedure for scaffolding a new MLflow experiment with Domino job execution.
---

# New Experiment Setup

When the user wants to set up a new experiment:

1. Create a training script in /mnt/code/experiments/
2. Add MLflow tracking with experiment name matching the project
3. Commit and push the script
4. Run as a Domino job using run_domino_job
5. Check results and share the MLflow run URL

When a user asks "set up a new experiment," Cursor evaluates the description, loads the rule, and follows the workflow steps. This provides similar functionality to Claude Code's slash commands, though activation depends on the AI's judgment rather than an explicit command.

Team-Wide Rule Distribution

For organizations on Cursor's Team or Enterprise plans, use the Cursor dashboard to configure Team Rules that are enforced across all team members. These take highest precedence and cannot be overridden by project or user rules. This is useful for organization-wide compliance requirements.

For teams not on enterprise plans, maintain a shared Git repository of .mdc files and copy them into each project's .cursor/rules/ directory. You can automate this in the pre-run script: 

# Copy org-wide rules from imported repo into .cursor/rules/
ORG_RULES="/mnt/imported/code/org-cursor-rules"

if [ -d "$ORG_RULES" ]; then
    mkdir -p /mnt/code/.cursor/rules
    for rule_file in "$ORG_RULES"/rules/*.mdc; do
        [ -f "$rule_file" ] || continue
        cp "$rule_file" "/mnt/code/.cursor/rules/$(basename "$rule_file")"
    done
    echo "[ok] Copied $(ls "$ORG_RULES"/rules/*.mdc 2>/dev/null | wc -l) org rules"
fi

Verification

To verify the setup is working:

  1. Create a new project from the template in Domino.
  2. Start a workspace (any IDE; Cursor connects remotely).
  3. Open the project folder in Cursor on your local machine.
  4. Switch to Agent mode (Cmd+L on macOS, Ctrl+L on Windows/Linux).
  5. Ask: "What MCP tools do you have available?"

Cursor should list the domino_server tools (run_domino_job, check_domino_job_run_status, etc.). If it does, the MCP connection is working. Try a follow-up: "List the files in my Domino project" to confirm end-to-end connectivity.

If the tools do not appear, check:

  • .cursor/mcp.json exists and is valid JSON.
  • The MCP server dependencies (fastmcp, requests, python-dotenv) are installed in the Domino environment.
  • The imported plugin repo is mounted at /mnt/imported/code/domino-data-lab-plugin.
  • Environment variables DOMINO_USER_API_KEY and DOMINO_API_HOST are set in the workspace.

To verify rules are loading:

  1. Open Cursor Settings and check the Rules section to confirm project rules are detected.
  2. Ask: "What rules do you have loaded?"
  3. The AI should reference the Domino platform rules from domino.mdc.
