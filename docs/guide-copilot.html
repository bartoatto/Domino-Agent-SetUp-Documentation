Domino GitHub Copilot Setup Guide

A project template guide for configuring GitHub Copilot with Domino and your domain knowledge

Overview

GitHub Copilot is an AI coding assistant available as a VS Code extension (with agent mode) and as a CLI (gh copilot). In agent mode, Copilot can take actions on your behalf: editing files, running commands, querying external systems, and analyzing data. Out of the box, it has no knowledge of Domino, your internal tooling, or your domain.

This guide shows how to close that gap.

Why the Domino plugin matters

Without it, Copilot has no knowledge of how Domino works. The plugin gives Copilot everything it needs to operate on the platform: running jobs, accessing datasets, deploying apps, tracking experiments. You can tell it to run your training script as a Domino Job and it knows exactly what to do.

Why adding your own instructions makes it more powerful

The plugin covers the platform. Your instructions cover your domain. When you add instructions for your data pipelines, internal APIs, and compliance rules, Copilot stops being a generic assistant and starts understanding how your team specifically works.

Two layers, in short

LayerSourceWhat it covers
Domino platform pluginProvided by DominoJobs, environments, datasets, apps, model endpoints, MLflow, tracing, distributed computing. Delivered as an MCP server.
Your instructionsAdded by your teamYour workflows, data pipelines, internal APIs, compliance rules. Not included in the template. Explained in Part 4.

Important: Copilot MCP tools are only available in agent mode. You must select "Agent" in the Copilot Chat mode dropdown (VS Code 1.99+). MCP tools are invisible in Ask or Edit mode.

Part 1: How It Fits Together

The template ships with these files:

/mnt/code/ ← Your project (Git repo root in Domino) │ ├── .github/ │ ├── copilot-instructions.md ← Domino-specific rules (system prompt) │ └── instructions/ │ └── domino.instructions.md ← Path-specific Domino instructions │ ├── .vscode/ │ └── mcp.json ← Starts the Domino MCP server │ └── domino_project_settings.md ← Project name/user (read by MCP server)

.github/copilot-instructions.md is read at the start of every agent mode session. It tells Copilot what it is, what it can do, and how to behave in this workspace. Update it as your instructions evolve.

The Domino MCP server is started from the imported plugin repo. It is the same MCP server used by Claude Code, Codex, Cursor, and Kiro — the server code is agent-agnostic. Only the configuration file format differs.

/mnt/imported/code/domino-data-lab-plugin/ ← Shared plugin (not in your repo) ├── skills/ ← 20 Domino platform skills (Agent Skills standard) ├── commands/ ← Slash commands (Claude Code specific) ├── agents/ ← Specialized subagents (Claude Code specific) └── mcp-servers/ ← Domino MCP server (agent-agnostic)

Note: The plugin's skills, commands, and agents directories are designed for the Agent Skills standard (Claude Code, Codex, Kiro). Copilot uses its own .instructions.md format instead. The MCP server is directly reusable. For skills, you can convert the content into .instructions.md files — see Part 4.

Part 2: The Shared Domino Plugin

The plugin at /mnt/imported/code/domino-data-lab-plugin is sourced from dominodatalab/domino-claude-plugin. It provides 20 skills covering the full Domino platform: jobs, workspaces, environments, datasets, apps, model endpoints, MLflow, GenAI tracing, distributed computing, governance, and more.

It also bundles the domino_server MCP server, which gives Copilot runtime tools:

  • run_domino_job — run a script as a Domino job
  • check_domino_job_run_status / check_domino_job_run_results — poll and retrieve job output
  • list_domino_project_files — browse project files
  • sync_local_file_to_domino / smart_sync_file / upload_file_to_domino_project / download_file_from_domino_project — file sync
  • get_domino_environment_info — inspect compute environments
  • open_web_browser — open Domino URLs

The MCP server is started via .vscode/mcp.json. It is wired up as part of the project template in Part 3.

Part 3: Setting Up the Copilot-Domino-Template and Environment

Two things need to be created once and reused across all projects: a Domino environment and a Domino project template. The environment provides the runtime: system utilities and MCP server dependencies. The template provides the project structure: the configuration files and imported repositories that Copilot needs to connect to Domino. When a team member creates a new project, they select both; everything comes pre-loaded.

Part 3a: Setting Up the Environment

GitHub Copilot is a VS Code extension — it does not need to be installed in the Domino environment. The environment only needs system utilities and MCP server dependencies.

Step 1: Dockerfile Instructions

Create a new Domino environment based on any Debian/Ubuntu-based Python image and add the following Dockerfile instructions:

# —— Install system utilities ——
USER root
RUN apt-get update && \
    apt-get install -y ripgrep jq vim && \
    rm -rf /var/lib/apt/lists/*

# —— Install MCP server dependencies (required) ——
RUN pip install --no-cache-dir fastmcp requests python-dotenv

# —— Install additional packages (optional — add what your team needs) ——
RUN pip install dash dash-cytoscape ydata_profiling streamlit st-pages \
    streamlit-extras pandas pyarrow requests tenacity python-dateutil \
    gym torch jupyter-ai[all] jupyter-ai-magics[all] h2o prophet optuna \
    tensorflow folium streamlit-folium rfc3339
RUN pip install --upgrade dominodatalab-data snowflake-connector-python pyarrow
USER ubuntu

Base image requirements: The base image must be Debian/Ubuntu-based (uses apt-get and the ubuntu user). All Domino standard environments meet this requirement.

Users need: VS Code 1.99+ with the GitHub Copilot extension and a Copilot subscription (Individual, Business, or Enterprise). Agent mode must be selected in the Copilot Chat dropdown.

Step 2: Add the Pre-Run Script

The pre-run script runs automatically at every workspace start. For Copilot, it only needs to populate domino_project_settings.md from environment variables — no agent installation or skill symlinking required.

#!/bin/bash
# start_workspace.sh — Domino pre-run script for GitHub Copilot
# Populates project settings from environment variables.
# Run once automatically at workspace start.

set -e

echo "=== Copilot Workspace Setup ==="

# —— Populate domino_project_settings.md from environment ——
if [ -f "/mnt/code/domino_project_settings.md" ] && \
   [ -n "$DOMINO_PROJECT_NAME" ] && [ -n "$DOMINO_USER_NAME" ]; then
    cat > /mnt/code/domino_project_settings.md << EOF
# Domino project settings to use with the MCP server domino_server
project_name="${DOMINO_PROJECT_NAME}"
user_name="${DOMINO_USER_NAME}"
EOF
    echo "[ok] domino_project_settings.md populated from environment"
fi

echo "=== Copilot workspace setup complete ==="

Part 3b: Setting Up the Template

The template is a Domino project template backed by a git repository. The repository contains the configuration files that wire Copilot to the Domino platform. An additional repository is attached as Imported Code; it mounts at a fixed path that the MCP config references.

Step 1: Create the Git Repository

Create a new git repository containing these files:

/mnt/code/ ├── .github/ │ ├── copilot-instructions.md │ └── instructions/ │ └── domino.instructions.md ├── .vscode/ │ └── mcp.json └── domino_project_settings.md

Here is what each file should contain:

.vscode/mcp.json

Starts the Domino MCP server from the imported repo. Important: VS Code uses servers as the root key, not mcpServers. This is the most common setup mistake — copy-pasting a Claude Code or Cursor config without changing the key means the server will not load.

{
  "servers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

.github/copilot-instructions.md

The persistent system prompt for Copilot, read automatically at every agent mode session. Copilot instructions are plain Markdown with no frontmatter. Start with the core Domino platform rules below and extend with your organization’s requirements.

# Domino Platform Instructions

You are a Domino Data Lab powered agentic tool that helps run tasks
and workflows on the Domino Data Lab platform on behalf of the user
using available tool functions provided by the domino_server MCP server.

## Domino Project Settings

Before making any MCP tool calls, read `domino_project_settings.md` to get
the `project_name` and `user_name`. If they contain placeholder values,
immediately update the file with the real values from environment variables:
- Project name: `$DOMINO_PROJECT_NAME`
- Username: `$DOMINO_USER_NAME`

## MCP Server

The `domino_server` MCP is configured in `.vscode/mcp.json`. Available tools:
- `run_domino_job` — run a script as a Domino job
- `check_domino_job_run_status` / `check_domino_job_run_results` — poll and retrieve job output
- `list_domino_project_files` — browse project files
- `sync_local_file_to_domino` / `smart_sync_file` / `upload_file_to_domino_project` / `download_file_from_domino_project` — file sync
- `get_domino_environment_info` — inspect compute environments
- `open_web_browser` — open Domino URLs

## Core Behavior

- For training or data extraction tasks, run commands as Domino jobs rather
  than in the local terminal. This ensures governance, reproducibility,
  and audit trails.
- When running a job, always check its status and results if completed,
  and briefly explain any conclusions from the result of the job run.
- If a job result includes an MLflow or experiment run URL, share that
  URL with the user.

## Data Access

- Project data is accessible under /mnt/data/{PROJECT_NAME}/ (local
  dataset) or /mnt/imported/data/ (shared/imported datasets).
- Before using a dataset, run a job to list all folder contents
  recursively to understand the full path to dataset files.
- Always create scripts to understand and transform data via job runs.
- Analytical outputs should be in plain text tabular format sent to
  stdout for easy result checking.

## Git Discipline

- Always check for uncommitted changes before running Domino jobs.
- You must commit and push changes before running any Domino jobs,
  because Domino executes against the remote repository state.

## App Development

- When building Domino apps, bind to 0.0.0.0 (not localhost).
- Use base: './' in Vite config for React apps behind Domino proxy.
- Create an app.sh as the launch file entry point.

.github/instructions/domino.instructions.md (optional)

Path-specific instructions that apply only when working with Python files. Uses applyTo frontmatter with glob syntax:

---
applyTo: "**/*.py"
---

When writing Python scripts that will run as Domino jobs:
- Include `#!/usr/bin/env python3` as the first line
- Print results to stdout in plain text tabular format
- Use `/mnt/data/` paths for dataset access, not relative paths
- Log experiment metrics with MLflow when doing model training

domino_project_settings.md

Tells the MCP server which Domino project to operate against. Auto-populated at workspace start by the pre-run script. Users do not need to edit it.

# Domino project settings to use with the MCP server domino_server
project_name="$DOMINO_PROJECT_NAME"
user_name="$DOMINO_USER_NAME"

Step 2: Register as a Domino Project Template

In Domino, create a new project template pointing to the git repository you just created. When configuring the template, attach the following repository as Imported Code:

RepositoryMounts at
https://github.com/dominodatalab/domino-claude-plugin.git/mnt/imported/code/domino-data-lab-plugin

These are the paths that .vscode/mcp.json references. Also set the default environment to the one built in Part 3a and the default hardware tier (Small is sufficient).

Step 3: Verify

Create a new project from the template, start a workspace, open VS Code with the Copilot extension, switch to Agent mode, and try: “What MCP tools do you have available?”

Copilot should list the tools from the domino_server MCP server. If it does, the template is working correctly.

Gotcha: The first time Copilot uses an MCP server, it asks for trust confirmation. Accept it once and it persists.

Part 4: Building Your Domain Instructions

Out of the box, the template gives Copilot knowledge of the Domino platform only. The domain layer (your workflows, your data, your internal APIs) is empty. This is where your team’s work begins.

Copilot’s Instructions System

Copilot supports three levels of custom instructions, layered together:

LevelSourcePriority
PersonalVS Code settings or ~/.copilot/Highest
Repository.github/copilot-instructions.mdMedium
OrganizationGitHub org settingsLowest

When multiple sources exist, they are combined (not overridden).

Path-Specific Instructions

For domain-specific knowledge that applies only to certain files, create .instructions.md files in .github/instructions/:

---
applyTo: "pipelines/**/*.py"
---

# ETL Pipeline Rules

## Overview
Rules for running ETL pipelines on Domino using Spark and the data warehouse.

## Key tools
Use the domino_server MCP tools to run pipeline scripts as Domino jobs.

## How to run a pipeline
1. Write the pipeline script in pipelines/
2. Commit and push
3. Use run_domino_job to execute it

## Rules
- All pipelines read from /mnt/imported/data/warehouse/
- Output to /mnt/data/{PROJECT_NAME}/pipeline-output/
- Log metrics with MLflow

Multiple patterns can be specified with commas: applyTo: "**/*.ts,**/*.tsx".

Where to Put Instructions

There are two scopes:

Project-specific instructions: add to .github/instructions/ in your project git repo. They persist as long as you commit and push.

Org-wide instructions: configure in GitHub.com under Organization settings > Copilot > Custom instructions. These apply across all repositories for all organization members. Good for org-wide Domino conventions that every project should follow.

Converting Agent Skills to Copilot Instructions

If you have skills in the Agent Skills standard (SKILL.md format, used by Claude Code, Codex, and Kiro), you can convert them to Copilot’s .instructions.md format:

An Agent Skills SKILL.md:

---
name: etl-pipeline
description: Rules for running ETL pipelines on Domino.
---
[body content]

Becomes a Copilot .instructions.md:

---
applyTo: "pipelines/**/*.py"
---
[same body content]

The name and description fields are dropped. Copilot uses applyTo glob patterns instead of description-based matching to decide when instructions apply.

copilot-instructions.md vs. .instructions.md files

.github/copilot-instructions.md: universal rules for every conversation — platform behavior, path conventions, compliance.

.github/instructions/*.instructions.md: context-specific knowledge for particular file types or workflows. If a rule only applies when working with a specific pipeline or tool, it belongs in an instructions file.

Part 5: The Complete Template

Once the template is set up and the pre-run script has run, a fully built-out project looks like this:

/mnt/code/ ← Your project (Git repo root in Domino) ├── domino_project_settings.md ← project_name and user_name ├── .github/ │ ├── copilot-instructions.md ← Platform rules, data paths, compliance │ └── instructions/ │ ├── domino.instructions.md ← Domino conventions for Python files │ ├── etl-pipeline.instructions.md ← your workflow │ ├── data-sources.instructions.md ← your data access patterns │ ├── internal-apis.instructions.md ← your proprietary tools and APIs │ └── compliance.instructions.md ← your governance and regulatory rules └── .vscode/ └── mcp.json ← MCP server config

Part 6: Extending Further

Custom Agents

Copilot supports custom agent profiles in .github/agents/*.md. These define specialized personas with specific tool access:

---
name: domino-assistant
description: Helps run Domino jobs and manage experiments
tools:
  - "domino_server/*"
---

You are a Domino platform assistant. Use the domino_server MCP tools
to run jobs, check status, and manage project files. Always run
training scripts as Domino jobs, not in the terminal.

Custom agents appear as selectable options in Copilot Chat. They can restrict which tools are available and set specific behavioral instructions.

Custom MCP Servers

Most teams will not need this initially; instructions cover a wide range of use cases. Write a custom MCP server when you need Copilot to take actions against an internal system at runtime: querying a proprietary database, submitting to an internal API, fetching from a data catalog.

A minimal MCP server using FastMCP:

from fastmcp import FastMCP
import requests, os

mcp = FastMCP("server-1")

def _auth():
    if os.environ.get("DOMINO_API_HOST"):  # inside a Domino workspace
        token = requests.get(
            "http://localhost:8899/access-token", timeout=5
        ).text.strip()
        return {"Authorization": f"Bearer {token}"}
    return {"X-Domino-Api-Key": os.environ.get("DOMINO_API_KEY", "")}

@mcp.tool()
def tool_1(query: str) -> dict:
    """Query internal-system-1. Returns matching results with locations."""
    resp = requests.post(
        "https://internal-system-1.myorg.net/api/search",
        json={"query": query},
        headers=_auth(),
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()

Register it in .vscode/mcp.json alongside the Domino server. Remember to use the servers root key:

{
  "servers": {
    "domino_server": {
      "command": "python",
      "args": ["/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    },
    "server_1": {
      "command": "python",
      "args": [".vscode/mcp-servers/server_1.py"]
    }
  }
}

Key principles: one tool per action, precise docstrings (Copilot reads them to decide when to call the tool), return structured data, include clickable URLs in responses when your tool creates something with a UI.

Copilot CLI

For terminal-based workflows, the Copilot CLI stores MCP configuration at ~/.copilot/mcp-config.json. Note that the CLI uses mcpServers as the root key (matching Claude Code and Cursor), not servers:

{
  "mcpServers": {
    "domino_server": {
      "command": "python",
      "args": [
        "/mnt/imported/code/domino-data-lab-plugin/mcp-servers/domino_mcp_server/domino_mcp_server.py"
      ],
      "env": {
        "DOMINO_API_KEY": "${DOMINO_USER_API_KEY}",
        "DOMINO_HOST": "${DOMINO_API_HOST}"
      }
    }
  }
}

You can also add MCP servers interactively with /mcp add in the CLI.

AGENTS.md Support (Experimental)

VS Code Copilot has experimental support for reading AGENTS.md (enable via chat.useAgentsMdFile setting) and CLAUDE.md (enable via chat.useClaudeMdFile). If you already have these files in your repo for other agents, Copilot can read them too — but these settings are opt-in and not yet stable.

Organization-Level Instructions

GitHub organizations can configure Copilot instructions that apply across all repositories for all members. This is a good place for org-wide Domino conventions. Configure in GitHub.com under Organization settings > Copilot > Custom instructions.
